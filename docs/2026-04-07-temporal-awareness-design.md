# Coach 时间感知与主动干预设计

**日期**：2026-04-07
**状态**：设计定稿，待实现
**前置依赖**：`docs/user-research/01-user-research-synthesis.md`（用户调研综合报告）

---

## 一、问题陈述

用户调研（2026-04-06，跨 4 平台约 35 个样本）揭示了一个高置信度的核心发现：

> **减脂失败的根因不是知识缺乏，而是关键时刻执行力缺失。**

"关键时刻"具有两个特征：

1. **可预测性**：节假日、聚餐、出差、搬家、感情变化、家庭事件等生活事件是系统性触发器，不是随机失败。
2. **不可自愈性**：用户在冲动时刻不会主动求助（洞察 2），失控后的自我攻击会加速下一轮失控（洞察 4）。

当前 Voliti 的 Coach 具备回顾性能力（事件文件记录已发生的行为），但缺乏前瞻性能力（不知道即将发生什么）。这限制了 Coach 的干预质量——一个不知道"明天你要搬家"的 Coach，和一个知道的 Coach，给出的干预是不同量级的。

同时，Coach 目前只有即时视角（单次对话中的状态感知与干预），缺乏长期旅程视角（跨周/跨月的模式识别与身份叙事维护）。调研中 6-8 周崩溃节点和"永久警惕疲劳"模式表明：用户不是在某一天突然放弃的，而是在身份认同缓慢侵蚀的过程中逐渐失去动力的。

## 二、设计原则

以下原则源自调研数据与讨论共识，所有后续设计决策必须与之一致。

### P1. 陪伴优先，工具其次

产品的核心竞争力是陪伴关系与共同经历，不是外部方法论或多巴胺驱动。Coach 的所有行为应体现内在动机导向：关注用户的心理状态与身份一致性，而非外部指标与行为纠正。

### P2. 渐进、授权、有据

主动干预必须满足三个条件：
- **渐进**：从轻量提问逐步深入，不突然介入深层话题
- **授权**：在用户知情、鼓励或默许的前提下进行
- **有据**：有充分的历史模式数据支撑，而非基于日历标记的机械触发

### P3. 信息采集边界由 Coach 判断

不同用户有不同的隐私偏好，不同文化背景下的边界感也不同。系统不预设"哪些生活事件可以问，哪些不可以"，而是由 Coach 在对话中根据用户的开放程度自行判断。

### P4. 内生建构，非预设框架

用户与 Coach 的对话内容应是自然涌现的，不由系统预设话题列表。系统的职责是为 Coach 提供足够的上下文和工具，使其能够在对话中做出高质量的判断。

## 三、统一时间轴模型

### 3.1 概念定义

统一时间轴是 Coach 对用户生活的时间感知，由三类条目组成：

| 条目类型 | 时间方向 | 来源 | 示例 |
|---------|---------|------|------|
| **行为记录** | 回顾性 | 用户报告 + Coach 记录 | "周三中午吃了火锅，情绪低落" |
| **生活事件** | 回顾性 | 用户在对话中提及 | "昨天和女朋友吵架了" |
| **前瞻标记** | 前瞻性 | 用户在对话中提及 | "下周六朋友聚餐"、"下月初搬家" |

行为记录对应现有的事件文件系统（Do 阶段）。生活事件和前瞻标记是新增的信息类别。

### 3.2 与现有系统的关系

统一时间轴不是新建的独立系统，而是对现有 memory 机制的扩展：

- 行为记录 → 已有的事件文件（保持不变）
- 生活事件 + 前瞻标记 → Coach memory 中新增的时间标记区域

存储位置建议在现有的 `/user/` memory 命名空间下，具体路径待实现时确定。

### 3.3 信息采集方式

所有时间轴信息均来自 Coach 与用户的自然对话，不依赖外部系统集成。采集时机包括：

- 晨间 Check-in 中的自然询问（"今天有什么特别的安排吗"）
- 用户主动提及的任何生活事件
- 晚间复盘中用户回顾的当日经历
- 对话中 Coach 基于自身判断做出的追问

节假日信息作为例外，可在 Coach system prompt 中基于用户 locale 注入公共节假日作为背景参考，但不作为干预的直接触发条件。

## 四、双时间视角

### 4.1 即时视角（已有，需增强）

**定义**：Coach 在单次对话中对用户当前状态的感知与干预。

**已有能力**：
- 晨间 Check-in → State 感知
- LifeSign 情境匹配 → 微干预触发
- 晚间复盘 → Check 与 Act

**增强方向**：在每次对话中，Coach 可访问前瞻标记，使即时干预具备上下文感知。例如：用户晨间 Check-in 报告"状态不错"，但 Coach 知道今晚有聚餐标记，可以在对话中自然地引入相关预案讨论。

### 4.2 长期旅程视角（新增）

**定义**：Coach 跨会话地追踪用户行为模式演变与身份叙事变化的能力。

**实现机制**：

通过 LangGraph Middleware 实现一个定期运行的后台分析脚本（建议频率：每 3-4 天或每周一次），执行以下逻辑：

1. **回顾扫描**：分析用户近期时间轴上的行为记录与生活事件，识别重复模式（如"聚餐后连续 N 天 Check-in 缺失"）
2. **前瞻扫描**：检查前瞻标记中即将到来的事件，评估与历史模式的关联
3. **摘要注入**：将分析结果以结构化摘要的形式注入下一次 Coach 会话的 prompt 中

Coach 收到摘要后可自行决定是否在对话中使用这些信息。如果分析结果无显著发现，Coach 可以忽略。

**长期视角的两个核心产出**：

- **模式洞察**：帮助用户看到自己的重复触发链（"你注意到了吗，最近三次聚餐后你都……"）
- **身份叙事更新**：在关键时间节点（特别是 6-8 周区间）帮助用户重新定义"我是谁"——"你已经不是六周前的那个人了"。可与签名体验（Signature Experience）联动，在旅程节点生成新的视觉锚定。

## 五、主动干预机制

### 5.1 干预通道

| 通道 | 触发条件 | 上下文需求 |
|------|---------|-----------|
| **会话内干预** | 用户打开 App 并与 Coach 对话 | Coach memory + 时间轴 + 长期视角摘要 |
| **Push 通知干预** | 后台脚本或时间触发 | Coach memory（人格一致性，不需要完整对话上下文） |

### 5.2 Push 通知的设计约束

- **人格一致性**：Push 消息中的 Coach 与对话中的 Coach 是同一个"人"，语气与个性化程度必须一致。Coach 自身的 memory（AGENTS.md + context.md）已包含足够的用户画像信息来支撑 push 内容生成，不需要加载完整会话上下文。
- **置信度门槛**：Push 干预不是"日历上有标记就发"，而是基于历史模式数据判断"这类事件对这个用户是高风险的"后才触发。
- **渐进性**：初期 push 频率应保守，随用户的正向反馈逐步增加。

### 5.3 干预内容的哲学定位

所有干预内容应体现陪伴关系与内在动机导向：

- **正确**："明天聚餐，你上次提到这种场景容易失控。想聊聊吗？"
- **错误**："明天聚餐，记得控制饮食！"

前者是陪伴者的关心，后者是提醒系统的指令。差异在于 Coach 是否将主动权交还给用户。

## 六、Onboarding 冷启动

### 6.1 问题

新用户的时间轴为空，Coach 没有历史模式数据，无法做个性化干预。

### 6.2 冷启动策略

在现有 Onboarding 必答层基础上，增加两个采集环节：

**环节 A：通用高风险场景认领**

基于调研数据中跨平台一致的高频触发场景，向用户展示 Top 3-5 常见失控场景（节假日、聚餐/社交、出差/差旅、情绪低谷、疲劳/睡眠不足），引导用户选出"对我来说最危险的"。

这一步的价值不仅是数据采集——它本身就是认知干预：让用户（特别是 Serial Dieter）在开始之前就看到"这些模式是普遍的，不是你的个人缺陷"。

**环节 B：近期前瞻事件采集**

自然地询问用户近期（未来 2-4 周）是否有已知的特别安排。采集到的信息成为时间轴的首批前瞻标记。

这两个环节应融入现有的对话式 Onboarding 中，保持轻量。如果用户不愿展开，属于后续补采范围。

## 七、用户与 Coach 的日常对话内容

### 7.1 设计立场

用户与 Coach 的对话内容是内生建构的，不由系统预设。以下分类是对"用户自然会聊什么"的观察性总结，供 Coach 设计参考，不作为对话框架强制执行。

### 7.2 五类自然话题

| 话题类别 | 本质 | 时间方向 | Coach 应如何处理 |
|---------|------|---------|----------------|
| 饮食情况 | 行为记录 | 回顾性 | 记入事件文件，关注情境而非数字 |
| 个人状态与心情 | 状态采样 | 当下 | State 评估，决定当日策略基调 |
| 行程与生活安排 | 环境预判 | 前瞻性 | 记入前瞻标记，关联已有 LifeSign |
| 身体数据变化 | 趋势信号 | 回顾性 | 放入模式识别上下文，避免单点解读 |
| 情绪波动（喜悦与低落） | 触发器识别 | 当下 + 回顾 | 记录情绪事件，识别触发模式 |

### 7.3 与三节奏的关系

三节奏（晨间 Check-in / 日间微干预 / 晚间复盘）是 Coach 的结构性动作框架，五类话题是用户自然带入的内容。两者正交：

- 晨间 Check-in 可能涉及状态、行程、情绪
- 日间微干预可能基于行程标记或情绪触发
- 晚间复盘可能涉及饮食、情绪、身体数据

三节奏决定"什么时候聊"，话题内容决定"聊什么"。Coach 在每个触点中根据用户状态和时间轴上下文自行决定话题优先级。

## 八、6-8 周危机的系统性应对

### 8.1 现象

调研数据一致表明：第 6-8 周是可预测的高流失风险期。三者叠加形成"完美风暴"：新鲜感消退、目标仍远、外部反馈减少。

### 8.2 应对策略

6-8 周危机不作为独立功能设计，而是长期旅程视角的自然产出。**关键设计约束：干预基于个人行为信号检测，而非固定时间窗口触发。** 6-8 周是调研中的统计参考区间，但不同用户的危机节点可能出现在第 4 周或第 12 周。JourneyAnalysisMiddleware 应检测以下个人化信号，不绑定固定周数：

1. **行为信号检测**：分析脚本持续关注早期衰减信号（Check-in 频率下降、情绪评分波动、LifeSign 激活率降低），当信号达到阈值时触发，不依赖"第 N 周"判断
2. **预防性对话**：将检测结果注入 Coach prompt，Coach 在对话中自然地引入身份叙事更新，而非"你到第 6 周了要加油"
3. **签名体验联动**：在这个时间窗口触发旅程节点的签名图像生成，帮助用户视觉化地锚定"我已经走了多远"
4. **非惩罚性响应**：如果用户在此期间出现失控，Coach 的第一句话是"最近怎么样"，不是"你上次没 Check-in"

### 8.3 关键认知

6-8 周危机的本质不是意志力不足，而是**身份认同的侵蚀**——"我是一个在减脂的人"这个标签在持续摩擦下磨损到无法支撑日常决策。Coach 的长期视角应聚焦于帮助用户持续更新这个身份叙事，使其从"正在减脂"演化为"在复杂环境中仍能做出清晰选择的人"。

## 九、实现优先级

以下按可行性与影响力排序，供代码重组参考：

| 优先级 | 模块 | 内容 | 依赖 |
|-------|------|------|------|
| **P0** | Coach Memory 扩展 | 在现有 memory 命名空间中增加时间标记区域（前瞻标记 + 生活事件） | 现有 CompositeBackend |
| **P0** | Coach 对话能力增强 | 更新 Coach system prompt，使其具备行程感知采集与前瞻标记写入能力 | P0 memory 扩展 |
| **P1** | Onboarding 冷启动 | 新增通用高风险场景认领 + 近期前瞻事件采集环节 | P0 memory 扩展 |
| **P1** | 长期视角 Middleware | 实现定期后台分析脚本，输出结构化摘要注入 Coach prompt | P0 memory 扩展 + LangGraph Middleware |
| **P2** | Push 通知基础设施 | iOS 端 push notification 能力 + 后端 push 触发机制 | iOS 客户端改造 |
| **P2** | Push 内容生成 | 基于 Coach memory 生成人格一致的 push 消息 | P2 push 基础设施 + P1 长期视角 |
| **P2** | 签名体验联动 | 在长期视角检测到关键时间窗口时触发签名图像生成 | P1 长期视角 + 现有 Intervention Composer |

## 十、开放问题

以下问题在本次设计讨论中未完全解决，需要在实现过程中进一步明确：

1. ~~**时间标记的存储格式**~~：**已决策** — 使用结构化 JSON schema（CEO Review 2026-04-07）。
2. **后台分析脚本的模型选择**：长期视角的分析任务是否使用与 Coach 相同的模型（gpt-5.4），还是使用更轻量的模型（gpt-5.4-nano）以降低成本。
3. **Push 通知的频率与退出机制**：用户如何控制 push 频率，是否需要在 Onboarding 中设置偏好，还是完全由 Coach 自适应。
4. ~~**"场景认领"的具体交互设计**~~：**已决策** — 使用 A2UI FanOutPanel 结构化多选组件（CEO Review 2026-04-07）。
5. **JourneyAnalysisMiddleware 触发机制**：**已决策** — 会话触发 + 时间门槛（每次会话开始时检查上次分析时间，超过 3 天才触发）。
6. **标记文件过期与压缩策略**：当标记文件增长过大时的压缩/归档机制，待实现时评估复杂度。
7. **前瞻标记的可变性管理**：事件取消、延期等变更的更正闭环（Codex Outside Voice 发现）。

# 生成式UI与Agent原生前端设计

## 一句话简介

当 LLM 不仅生成内容而且生成界面本身时，AI 产品的前端交互范式会发生根本转变：[[Google Research]] 证明 LLM 生成 UI 的可用性（82.8% 偏好率），[[Google Developers]] A2UI 协议定义了声明式安全路径，[[Jakob Nielsen]] 划定了适用边界——三者共同构成 2026 年 Agent-Native 前端设计的核心知识框架。

## Takeaway

1. **Generative UI 已从概念验证走向实际可用** — 82.8% 人类偏好率（vs Markdown），但仅限中等复杂度以下任务
2. **两条路径的根本权衡是表达力 vs 可控性** — Full Generation 极具表达力但有安全和一致性问题；A2UI 结构性安全但受组件目录约束
3. **涌现性意味着这是模型能力驱动的趋势** — Gemini 3/2.5 Pro 错误率 0%，而 Gemini 2.0 Flash-Lite 错误率 60%；能力质变非工程积累
4. **Nielsen 的适用边界值得重视** — 对长期运行的复杂系统，纯 Generative UI 不适用，需更结构化方案
5. **核心问题已从"能不能做"变为"选哪条路径"和"如何组合"**

## 摘要

### Generative UI 论文：LLM 是有效的 UI 生成器

**核心命题**：Generative UI 是一种新模态——AI 模型不仅生成内容，而且生成整个用户界面。经适当提示与工具配备，现代 LLM 能为几乎任何 prompt 稳健地生成高质量定制 UI。（Leviathan et al., 2025）

> "LLMs transformed the world's finite collections of texts to an infinite collection... we are excited about a future where users don't have to pick from a finite library of applications or visual pages, but instead, they have access to an infinite catalog, where the right ephemeral interface is generated on the spot tailored for their need."

**系统三大组件**：（1）工具服务器——暴露图片生成、搜索等端点；（2）精心设计的系统指令——五层结构（核心理念→示例→规划指引→技术规格→动态信息），约 3K 词；（3）后处理器——9 类轻量组件，修复常见 JS/CSS/HTML 错误。

**核心设计哲学**：即使简单查询也优先构建交互应用而非静态文本（"Build Interactive Apps First"）；绝对禁止占位符或虚假数据；涉及具体实体时必须搜索验证。

**量化评估结果**（100 prompts from LMArena，成对人类偏好评估）：

| 格式 | ELO 分数 |
|------|---------|
| 人类专家网站 | 1756.0 |
| Generative UI | 1710.7 |
| Markdown UI | 1459.6 |
| 搜索引擎首位 | 1355.1 |
| 纯文本 | 1218.6 |

- vs Markdown：**82.8%** 偏好 Generative UI；信息检索类 prompt：**92.6%**
- vs 人类专家：**43%** win（56% expert win），44% 至少可比

**涌现性（Emergent Capability）**：旧模型（Gemini 2.0 Flash-Lite）错误率 60%，新模型（Gemini 3）错误率 0%。这是模型能力的涌现现象，而非工程技巧积累。

**论文局限**：生成速度慢（1-2 分钟，流式传输可减半）；评估仅涵盖相对简单的设计问题。

### Google A2UI：声明式 Agent-UI 协议

**本质**：声明式数据格式（JSON/JSONL），让 agent 从受信任组件目录中组合定制界面，只描述 UI 的**意图**，不描述实现。（Google Developers, 2025-12, Apache 2.0, v0.8）

> "Safe like data, but expressive like code."

**核心问题解决**：

| 传统方案 | 问题 | A2UI 方案 |
|---------|------|----------|
| 纯文本聊天 | 低效多轮对话 | Agent 直接生成表单 |
| Agent 生成 HTML/JS | 安全风险、视觉不一致 | 声明式 JSON，客户端原生渲染 |
| iframe 沙箱 | 笨重、视觉割裂 | 无需沙箱，数据即安全 |

**四阶段架构**：Generation（Agent 创建 JSON payload）→ Transport（协议无关传输）→ Resolution（验证 + 权限核验）→ Rendering（客户端用原生框架实现）。

**交互循环（Emit-Render-Signal-Reason）**：Agent 发送 JSONL → 客户端渲染原生组件 → 用户操作 → Renderer 发送**类型化 userAction 事件**（非自由文本）→ Agent 消费结构化信号决定下一步。类型化回传而非自由文本，是关键设计决策。

**安全模型**：客户端维护白名单式批准目录，Agent 只能请求目录内组件，从结构上消除 UI 注入风险。支持跨平台（Web/Flutter/SwiftUI）原生渲染同一 JSON。

**与 Generative UI 论文的本质区别**：Generative UI 追求极致表达力（可生成游戏/模拟器），A2UI 追求安全与可控（结构性安全，客户端完全控制渲染）。A2UI 是"说什么"，AG-UI 是"怎么传"——二者互补。

### Jakob Nielsen：Generative UI 的 UX 意涵与边界

核心观点（Nielsen, 2026）：

1. **从个性化到个体化** — 不是为用户群体调整界面，而是为每个用户的每个特定需求实时设计唯一界面
2. **"交互合成器"** — 消除传统网站的"导航税"（navigation tax），系统只呈现当前意图所需组件
3. **"更便宜的一次性 UI"** — UI 从服务百万用户多年的长期应用，变为按需创建的短暂交互

**适用边界**：适用于单次查询、信息检索类任务；不适用于复杂多步骤工作流、需要深层信息架构的系统（如 Excel、Netflix）。

**预测**：AI 将在 2026 年底在简单设计问题上超越人类设计师，AI UX 能力每年约提升 4 倍。

### 2026 年 Generative UI 生态全景

三种范式并存：

| 范式 | 代表 | 特点 |
|------|------|------|
| Full Generation | Google Generative UI | LLM 生成完整 HTML/CSS/JS，表达力最强，安全性最弱 |
| Declarative Protocol | Google A2UI | JSON 声明式，客户端原生渲染，安全性最强 |
| Hybrid/Framework | CopilotKit、Vercel AI SDK | 应用框架内集成 AI 生成组件，介于两者之间 |

关键趋势：Agent-native 界面设计正从实验阶段进入标准化阶段；实时流式更新和增量渲染成为基本要求；安全性（防 UI 注入）成为核心关注点。

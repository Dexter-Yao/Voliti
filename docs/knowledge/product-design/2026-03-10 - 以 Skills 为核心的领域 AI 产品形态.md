# 以 Skills 为核心的领域 AI 产品形态

## 一句话简介

当通用 Agent 泛化能力趋同，领域 AI 产品的竞争力来自"通用 Agent + 领域 Skill"的组合，以及围绕 Skill 建立的五层系统；Skill 同时作为效率工具与职业转型的方法论载体，是内容-社区-服务混合增长路径的核心引擎。

## Takeaway

1. **产品公式**：通用 Agent 提供推理与执行基础设施，领域 Skill 提供"做什么、怎么做好"的知识封装。但护城河不止于 Skill 文本——真正难以复制的是围绕 Skill 建立的五层系统：知识定义 + 工作流嵌入 + 上下文积累 + Eval 机制 + 反馈闭环。
2. **双重价值**：Skill 不只是效率工具，更是职业转型的方法论载体。用户在使用 Skill 做分析的过程中，逐步内化领域的分析范式——技能升级是使用的副产品，而非独立产品。这一机制使目标用户群体从"想降低成本的企业"扩展到"想完成职业转型的个人"。
3. **增长机制**：管理会计 Skill 的增长路径与消费应用的病毒传播逻辑不同，更接近 dbt 的专业社区模式——以内容示范建立可信度，以开源工具聚合同类从业者，以咨询服务完成深度变现。其中，能否在"AI 财务分析师"这一新职业身份上形成共识，是社群引力能否成立的核心前提。
4. **封装悖论**：Skill 文本无法成为持久壁垒，因为 prompt 没有编译屏障。护城河在执行层：经过大量真实案例校准的领域判断知识、可验证的交付记录，以及由此积累的客户信任。

## 为什么是 Skills

### 通用 Agent 时代的产品分化逻辑

2025 年后，通用 Agent 平台（Claude Code、ChatGPT、Cursor 等）的推理与执行能力已足够强大，绝大多数"怎么操作"的问题不再是瓶颈。真正的差距出现在"知道什么"——即领域知识的深度与准确性。

Anthropic 对 Skills 的官方定义是将"偏好、流程、领域知识"教给 Agent，用于可重复的工作流；Skills 由 instructions、scripts、resources 组成，构成可复用的任务包。OpenAI 在 harness engineering 的实践中同样强调：Repository knowledge 作为 system of record，真正稀缺的不是模型本身，而是"如何把知识、约束、验证和执行环境组织起来"。

Skills 恰好是在应用层承接这一问题的产品形态：它不替代 Agent，而是告诉 Agent"在这个领域，什么叫对、什么叫好、哪些边界不能越过"。[[2026-04-09 - Harness把执行轨迹变成智能飞轮]] 对这一点的补充很关键：skills 不应被孤立理解为一包静态提示词，而应被放回 harness 的信息层、执行层与反馈层中看待。只有当 skill 同时参与上下文筛选、任务拆解和错误复盘时，它才真正进入可积累的执行系统，而不是一次性的知识封装。

### Skill 作为方法论载体（不只是效率工具）

这是 Skills 产品形态一个被低估的维度：Skill 在帮用户完成任务的同时，也在传递完成这类任务的思维方式。

一个设计良好的管理会计 Skill 在运行时会：告诉用户应该提出哪些分析问题、数据应该如何组织、差异分析的结论如何归因、报告中哪些内容需要管理层关注。这些不只是 Agent 的执行步骤，也是管理会计的分析范式。

用户每次使用 Skill，都在被动习得这个范式。**技能升级是使用的副产品，而非产品功能本身**。这与 dbt（一款数据分析师广泛使用的开源数据转换工具，详见下文"新岗位范式"一节）的运作方式高度一致——数据从业者学 dbt 不是为了学习，而是因为工作需要；但学习是自然发生的。

## 护城河的五层系统

领域 AI 产品的护城河，不应被理解为单一的"Skill 文本质量"，而是五个层次的系统性积累，缺任何一层都难以构成持久优势：

**第一层：领域知识定义（Skill 本体）**
告诉 Agent 什么叫"正确的分析"、什么样的输出会误导管理层、哪些场景需要标注假设前提。这是护城河的起点，但也是最容易被模仿的一层。

**第二层：工作流嵌入**
Skill 必须嵌入用户实际工作场景。管理会计场景中，这意味着与 Excel 导出格式对接、与企业现有报告模板兼容、输出结果可以直接被管理层使用而无需二次加工。嵌入深度决定了替换成本。

**第三层：专有上下文积累**
每个客户的历史数据、KPI 定义、利润中心结构、业务术语——这些上下文随着合作深入逐步沉淀。对同一家企业，一年积累的上下文使 Skill 的输出质量与刚介入时不可同日而语。新竞争者无法复制这层积累。

**第四层：Eval 与校准机制**
谁定义"差异分析结论是否准确"的验收标准，谁就实际控制了 Agent 的行为边界。这与 [[2026-03-09 - OpenAI Harness Engineering Agent优先工程实验全景]] 中 harness engineering 的核心洞察一致：Eval 是隐藏护城河。在管理会计场景中，"合理的假设是什么"本身就是领域专业性的体现，无法从 Skill 文本中读取，只能从大量真实交付中学习。

**第五层：反馈闭环**
出了错如何发现？发现后如何修正？修正后如何防止重复出错？这套工程化的反馈机制，将每一次交付失误转化为系统改进的输入。没有这一层，前四层的积累会在时间中缓慢退化。

## 新岗位范式：Skill 作为职业转型的载体

这是整个机会中最容易被低估的维度。

dbt（data build tool）是一款面向数据分析师的开源数据转换工具，由 dbt Labs 开发。它允许分析师用 SQL 编写数据转换逻辑，并以软件工程的方式管理这些逻辑：版本控制、自动测试、文档化。截至 2025 年，dbt 已被超过 60,000 个企业团队采用，拥有 10 万人规模的活跃社区，是数据领域最成功的开源商业化案例之一。dbt Labs 同时提供免费的开源版本（dbt Core）和付费的企业云产品（dbt Cloud），变现发生在企业服务层而非工具本身。

**dbt 的真正成功密码**是它创造了一个新职业身份——analytics engineer，而不只是发布了一个好用的工具。analytics engineer 站在数据工程与业务分析的交叉点，dbt 定义了这个角色的工作方式：以代码管理数据转换、以版本控制追踪分析逻辑、以文档使数据可被他人理解。成千上万的数据从业者把"会用 dbt"写进简历，因为这代表一种新的专业能力。

管理会计 Skill 面对的是一个类似的历史机遇。中国超过 1,400 万财会人员中，超过 85% 集中在财务会计岗位——他们感受到职业压力，知道自己需要向管理会计方向转型，但缺乏清晰的路径。财政部推了十年管理会计改革，成效有限，根本原因之一是：准则有了，但没有人告诉财会人员"用这些准则，你应该做什么样的具体分析"。

一套设计良好的管理会计 Skill 恰好填补了这个空白：它不是培训课程，而是直接可操作的工作工具；用它做一次差异分析，用户就理解了差异分析应该怎么做。如果这个 Skill 能被成千上万的财务分析师用于实际工作，并且他们能把"用 AI 做管理会计分析"这个能力写进职业档案，那么这个 Skill 就不只是产品，而是一个新职业身份的定义者。

"AI 财务分析师"或"业财一体化分析师"能否成为一个被广泛认可的新职业身份，是这类产品能否形成真正社区引力的核心变量。这一身份一旦形成共识，工具的传播逻辑将从推广转为吸引——从业者会因职业发展需求主动寻找、学习并推广相关工具，而非被动接受营销触达。

## 增长机制：内容-社区-服务混合模型

管理会计 Skill 的增长路径与消费级应用的自发传播逻辑不同。使用场景以月度为主、购买决策属于高信任专业服务性质、从业者群体分散——这些特征决定了增长不能依赖强制社交裂变，而是要通过内容建立可信度、通过开源聚合职业共同体、再由咨询服务完成深度变现。

这一路径更接近 dbt 的社区驱动模式而非 Slack 的协作裂变。dbt 没有"必须拉对方加入"的强制性，而是凭借工具本身对职业能力的提升吸引从业者主动靠近，形成社区后再延伸出企业级产品与服务。

## 商业模式三层结构

| 层级 | 产品 | 定价 | 作用 |
|------|------|------|------|
| **第一层** | 开源通用版 Skill | 免费 | 分发、获客、建立信任、积累专业认知度 |
| **第二层** | 付费垂直版 Skill | 订阅或一次性授权 | 针对特定行业/规模的深度封装，包含更完整的场景覆盖与校准知识 |
| **第三层** | 咨询服务 + 深度 Skill 交付 | 项目制 | 为特定企业定制框架，同时交付经过校准的私有 Skill |

第三层是最重要的变现层，也是校准知识积累最快的层次。每一个咨询项目都在向 Skill 注入更多的边界条件、行业特殊性与验收标准——这些反哺到第一层和第二层产品，使其质量持续提升。

**海外市场**（外资企业在华、东南亚）：有限许可证（limited license）而非完全开源可能更合适。管理文化更重视知识产权的市场环境中，有结构的商业许可比完全开放更容易建立合规信任。

## 可见性与封装的矛盾

### 悖论的本质

Skill 的完整文本需要对 Agent 可见才能被执行，但可见意味着可被复制。与代码不同，prompt 没有编译或混淆屏障——任何有访问权限的用户都可以直接阅读并复制 Skill 的全部内容。

这使"靠 Skill 文本本身建立竞争壁垒"在结构上是困难的。

### 三类应对策略及其局限

**受控执行界面**：将 Skill 封装在 SaaS 产品中，保护文本不被看见。代价是放弃"运行在用户已有 Agent 上"的核心优势，等同于退化为传统 SaaS。

**有限演示策略（Live Demo）**：公开展示输入输出，不暴露完整结构。早期有效，但无法长期阻止竞争者反向工程。

**许可证约束**：通过 SSPL、BSL 等许可证限制商业使用。提供法律层面的一定保护，但执行成本高，中国的法律效力存在不确定性。

### 护城河在执行层，而非文本层

三类策略都是延缓手段。真正可持续的优势来自五层系统中的第三至五层——上下文积累、Eval 机制与反馈闭环——这些无法从 Skill 文本中读取，只能从大量真实交付中积累。

开源策略并不意味着放弃竞争壁垒，而是将护城河从"文本保护"转移到"执行能力的显性化"：谁能展示更多的真实交付案例、谁的 Skill 在实际使用中更稳定准确，谁就占据市场信任。这与软件行业"开源引流、企业服务变现"的成熟路径在逻辑上完全一致。

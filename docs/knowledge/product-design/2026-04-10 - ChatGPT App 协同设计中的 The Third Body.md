# ChatGPT App 协同设计中的 The Third Body

## 一句话简介

这篇文章提出了 [[ChatGPT Apps]] 的核心结构问题：当模型成为系统中的“The Third Body”时，产品设计的首要任务就从“组织用户与界面”转为“组织用户、widget、模型三方之间的上下文分配、可见性和协作边界”。

这篇笔记要回答的问题是：为什么许多传统 web 产品的默认经验在 [[ChatGPT Apps]] 中会失效，以及 “The Third Body” 如何重塑产品、界面与工程实现。

## Takeaway

1. [[ChatGPT Apps]] 不是把网页放进聊天框，而是在用户、widget、模型三方之间建立新的协作结构；上下文如何分配与暴露，决定了产品是否成立。
2. 传统 web 的若干默认优化会在这里失效，例如懒加载、仅由 UI 持有状态、过滤器优先于语言、把生产配置当作上线前细节。
3. 在这种环境里，界面不再只是展示层，而是模型理解用户意图的组成部分；界面的当前状态若不能被模型感知，对话就会与操作脱节。
4. 生产环境中的安全与边界配置需要被产品团队理解：`CORS` 主要限制网页是否能向其他域名发请求，`CSP` 主要限制页面允许加载、连接和嵌入哪些外部资源；在 [[ChatGPT Apps]] 的 iframe 沙箱里，后者往往更常成为上线后的真实约束。
5. 当一类问题在多个 app 中反复出现，最合理的做法不是继续手工处理，而是将经验沉淀为 framework、hook、模拟器、QA 检查和 skill，形成可复用的开发基础设施。

## 摘要

### 背景

作者来自 [[Alpic]]。在 [[OpenAI]] 发布 [[Apps SDK]] 之后，团队用约三个月时间开发了二十多个 [[ChatGPT Apps]]，服务于内部与客户场景，涵盖旅行、零售、SaaS 等 B2B 与 B2C 领域。文章的目的不是介绍某个 API，而是总结这类应用在真实构建过程中的共同难点。

作者提出的核心概念是 “The Third Body”。传统 web app 主要处理“用户 + UI”的关系，而 [[ChatGPT Apps]] 中模型也成为主动参与者。系统因此不再只是一个人使用一个界面，而是三方共同参与同一个状态系统。真正困难的地方不是让三方永久同步，而是决定什么信息应被谁看到、何时看到、以什么形式看到。

> “With traditional web apps, things were simple: you only had a user and a UI. In a ChatGPT app, a third body enters the system: the model.”

### 一、三方协同首先是上下文问题

作者将最早的教训归结为一个错误直觉：默认“所有状态都应该共享给所有参与者”。实践表明，这会同时带来性能问题和逻辑问题。

从性能角度看，widget 往往需要比模型更多的数据，例如图片、价格变体、预加载选项；若全部发送给模型，会增加 token 消耗、响应延迟和认知噪音。从逻辑角度看，一些场景天然要求信息不对称。文章用两个游戏场景说明这一点：有时模型需要知道答案但用户不能知道；有时用户需要看到答案而模型不能知道。由此得到的结论是：上下文不应默认共享，而应被刻意分配。

这一判断进一步影响了性能策略。传统 web 倾向于懒加载，尽量减轻首屏负担；但在 [[ChatGPT]] 环境中，每一次额外工具调用都伴随推理与安全沙箱延迟，因此“按需再取”常常意味着多一次等待。作者的经验是，许多场景里更合理的做法是提前在第一次工具响应中放入更多关键数据，让 widget 尽快完成初始化。

第三个紧接着出现的问题是模型可见性。用户在 widget 中完成了某项操作后，如果模型无法感知当前界面状态，那么对话层就会与操作层脱节。用户在视觉上已经“选中”了某个对象，但模型仍然停留在旧上下文中，后续回答就会失真。作者因此强调，界面状态中那些会影响模型理解用户当前意图的部分，必须稳定地回流到模型上下文中。

这三类问题合起来构成文章最重要的产品判断：[[ChatGPT Apps]] 的本质不是组件集成，而是三方上下文治理。

### 二、界面设计因此进入 AI 原生范式

在这种三方协同结构下，作者认为许多传统 UI 习惯需要被重新审视。

首先，同一个 widget 可能以内联、画中画（PiP）、全屏三种模式出现。不同模式意味着不同的空间占用、不同的遮挡关系和不同的操作节奏，因此 UI 不能被简单看作一块固定尺寸的网页区域。作者的经验是，简单交互适合 inline，复杂高密度场景适合 fullscreen，而在生成后仍需伴随对话持续使用的内容，则适合 PiP。

其次，嵌入式环境中的一致性比独立产品更重要。因为 widget 活在 [[ChatGPT]] 的宿主界面内，任何视觉、间距、图标或交互语法上的偏离，都会比在独立网站中更明显。作者因此强调以 [[OpenAI]] 的 UI Kit 和 design tokens 为基线，让 widget 看起来像宿主体验的延续，而不是一个被塞进来的外部页面。

再次，作者对传统 dashboard 式过滤器提出了明确批评。在 agentic UI 中，如果用户已经可以直接用自然语言表达意图，再要求其回到 checkbox、range slider、排序面板等控件，实际上是在让用户从新范式退回旧范式。作者因此主张以语言作为主要入口，把参数映射与选项理解更多交给模型处理。

最后，文件输入被提升为一等公民。文章认为，很多 richer interaction 应该从“用户已经拥有的文件、图片或资产”开始，而不是从表单开始。文件不只是附加材料，而是工作流的起点，这使得 [[ChatGPT Apps]] 能更自然地贴近真实任务。

### 三、生产环境的关键约束来自安全与边界配置

当应用从本地开发走向线上环境后，作者发现一类不同的问题开始主导成败，其中最典型的是 `CSP`。

若不具备 web 背景，可以把这两个概念区分为：

- `CORS`：主要约束“当前网页能不能向其他域名发请求”
- `CSP`：主要约束“当前页面允许加载哪些脚本、图片、字体，连接哪些域名，嵌入哪些 iframe，跳转到哪些外部地址”

如果说 `CORS` 更像“我能不能去别人家取东西”，那么 `CSP` 更像“我家允许哪些外来东西进来”。

作者说 “CSPs are the new CORS”，并不是说两者等价，而是说：在传统 web 开发里，团队经常最先遇到跨域请求问题；但在 [[ChatGPT Apps]] 这种被严格 iframe 沙箱包裹的环境中，更常导致“本地可用、线上失败”的，往往是 `CSP` 配置不完整。因为 [[OpenAI]] 将 app 放在严格隔离的双层 iframe 中，资源加载、API 连接、外部嵌入与跳转域名都必须显式声明。

除了 `CSP`，一些看似很小的 widget flags 也会决定系统行为边界。例如，哪些工具可以被模型调用，哪些能力 widget 自己可以触发，发布后的 “Open in App” 按钮会进入什么位置。这些设置不只是工程参数，而是产品控制权如何分配的一部分。

### 四、长期效率来自专门的开发底座

文章最后几条经验集中在开发工作流本身。作者团队发现，如果继续用普通 web 工具链去构建 [[ChatGPT Apps]]，开发体验会被严重拖慢。

原因之一是热重载并不能天然工作。由于资源缓存和转发机制的存在，传统前端环境中熟悉的 HMR 无法直接套用到 [[ChatGPT]] 宿主里。作者团队为此专门搭建了热更新能力，以缩短调整界面后看到真实效果的反馈时间。

作者还强调，不是所有测试都必须放进正式的 [[ChatGPT]] 宿主环境中。对于早期的状态逻辑、布局问题和日志调试，本地模拟器更适合承担快速反馈职责；而那些真正依赖模型交互、边界条件和宿主环境的验证，再留到真实环境里完成。移动端测试也应被纳入常规流程，而不是作为最后补充，因为 [[ChatGPT]] 本身就是跨设备入口。

在前端抽象上，作者没有因为 [[Apps SDK]] 提供的是底层接口，就完全放弃已有工程习惯。相反，他们主动把常见能力重新包装成更符合 [[React]] 习惯的 hook 与状态管理方式。文章想表达的不是某个具体 hook 的价值，而是一个长期成立的工程判断：当底层环境变新，团队往往仍然需要一层熟悉、稳定、可复用的抽象，来控制复杂度。

### 五、经验最终应被编码成可复用资产

文章最后收束到一个更高阶的开发观：如果某类问题会在多个 [[ChatGPT Apps]] 中反复出现，就不应该继续依赖团队记忆或临时补丁，而应该把经验沉淀为 framework、hook、热重载工具、本地模拟器、QA 检查与 skill。

作者因此将这些经验编码进开源 framework 和专用 [[Codex Skill]] 中，覆盖从构思、代码生成、本地联调到发布检查与部署辅助的完整流程。这里真正重要的不是具体工具名字，而是方法论：把反复踩坑得到的隐性知识，转成显性的工程资产。

### 结论

这篇文章最值得长期保留的判断是：一旦模型成为产品中的“The Third Body”，前端、状态、性能优化、过滤器、生产配置与开发工具链都需要围绕三方协作重写。最好的 [[ChatGPT Apps]] 不是传统产品的聊天外壳，而是围绕 AI-first interaction 重新长出来的新型体验。

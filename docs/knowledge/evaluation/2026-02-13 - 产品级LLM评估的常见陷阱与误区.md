# 产品级LLM评估的常见陷阱与误区

## 一句话简介

总结构建 LLM 产品时最常见的七类评估错误，基于一线 AI 公司生产实践与学术研究，聚焦应用层评估（产品级评估，Product-Specific Evals）而非模型基准测试（Foundation Model Benchmarks）。

## Takeaway

| 陷阱 | 核心教训 | 关键行动 |
|------|---------|---------|
| 无需求启动 | Error Analysis 先于代码 | 手工测试 20-30 个案例 |
| 混淆模型与产品 | 高基准 ≠ 好产品 | 测试完整用户旅程 + 自定义指标 |
| Vibe Testing（凭感觉测试） | 主观感受不可靠 | 50+ 自动化测试，量化标准 |
| 追求 100% 通过 | 挑战性不足 | 健康通过率 70-90% |
| 多维度混合评分 | 1-5 分制是错误的 | 每维度独立二元判断 |
| **Judge 偏见** | **κ 通常仅 0.1-0.3** | **位置随机化 + 人工校准（κ ≥ 0.7）** |
| 忽视生产监控 | 离线测试非穷尽 | 采样 5-20% + 闭环反馈 |

---

## 摘要

### 核心区分：两类评估的本质差异

| 维度 | Foundation Model Benchmarks | Product-Specific Evals |
|------|---------------------------|----------------------|
| 评估对象 | 基础模型能力（MMLU、HumanEval） | 产品是否解决用户问题 |
| 数据集 | 标准化基准 | 自定义数据集，反映实际用例 |
| 评估时机 | 模型选型阶段 | 开发、测试、生产全周期 |

> "虽然 LLM 基准帮助比较 LLM，但它们**不适合评估基于 LLM 的产品**" （Confident AI, 2024）

> "不成功的 AI 产品几乎总是共享一个根本原因：**未能创建稳健的评估系统**" （Hamel Husain, 2024）

### 陷阱 1：启动 MVP 前未明确需求

**问题核心**：团队最大错误是没有明确需求就启动开发，导致返工。（Braintrust, 2024）

**解决方案**：
1. **Error Analysis First**（Hamel Husain, 2024）：手工测试 20-30 个真实用例，记录失败模式，**再写 eval**
2. **需求明确化**：定义 3-5 个核心成功标准，为每个标准准备 5-10 个正反例
3. **Looking at Traces**：人工审查足够多 traces，建立对"好/坏"的直觉

### 陷阱 2：混淆模型性能与产品性能

**问题核心**：高基准分数（如 MMLU 95%）不保证产品在实际任务上的表现。（Braintrust, 2024）

**解决方案**：
- 测试**完整用户旅程**（输入解析→检索→生成→格式化），而非仅 LLM 调用
- 使用真实用户输入，而非理想化测试输入
- 自定义指标 > 通用指标：投资针对应用的特定质量维度

### 陷阱 3：Vibe Testing（凭感觉测试）

**问题核心**：依赖"看起来不错"的主观感受，无法量化和捕捉边界情况。（Google, 2024）

**解决方案**：
- 将"感觉好"转化为可测量标准（如"感觉相关" → "包含用户问题中的关键实体"）
- 创建 50+ 自动化测试（覆盖 happy path + edge cases）
- 每次变更自动运行全部测试，跟踪通过率历史趋势

### 陷阱 4：过度追求完美通过率

**问题核心**：100% 通过率意味着挑战性不足。（Hamel Husain, 2024）

**解决方案**：
- 健康通过率：70-90%
- 区分 **Must-Pass**（95%+，核心功能）和 **Stretch Goals**（50-70%，挑战性场景）
- 持续添加系统**当前无法通过**的测试，代表改进方向

### 陷阱 5：单个分数混合多个维度

> "**不要偏离二元 pass/fail 判断**。1-5 分制指标被认为是错误的" （Hamel Husain, 2024）

**解决方案**：

```
❌ 错误：整体质量 (1-5 分)

✅ 正确：每个维度独立二元判断
- is_factually_correct: true/false
- is_relevant_to_query: true/false
- has_appropriate_tone: true/false
```

**"Too Many Metrics" 陷阱**：开始时只创建 3-5 个最关键指标，避免"以防万一"添加指标。

### 陷阱 6：忽视 LLM-as-Judge 的系统性偏见

**三大系统性偏见**（ACL Anthology, 2025；arXiv, 2024）：

| 偏见类型 | 表现 | 缓解策略 |
|---------|------|---------|
| **位置偏见（Position Bias）** | 偏好提示中特定位置的答案 | 位置随机化；位置切换取平均 |
| **冗长偏见（Verbosity Bias）** | 偏好更长答案，即使不准确 | 明确要求"简洁优于冗长" |
| **自我增强偏见（Self-Enhancement Bias）** | 偏好同模型生成的文本 | 使用不同 Judge 模型交叉验证 |

**可靠性数据**（ACL Anthology, 2025）：
- 多语言场景 Fleiss' κ 平均 **0.1-0.32**（远低于可信阈值 0.8）
- 低资源语言和推理任务中一致性**严重下降**
- 特定领域成功案例：认知扭曲检测中 GPT-4 达到 Fleiss' κ=0.78

**正确配置方法**（OpenAI, 2024）：
- 使用判别式任务：二元分类 > 多选题 > 成对比较 > 评分（1-5 分）> 开放式生成
- 强 Judge（如 GPT-4.1）可达 **80%+ 人类一致性**
- Judge Prompt 中提供 2-3 个正反例（包含判定理由）

**Braintrust 要求**（Braintrust, 2024）：人工标注 30-50 个案例，计算 Judge 与人类的 Cohen's κ；**如果 κ < 0.7，迭代优化 Judge Prompt**。

### 陷阱 7：忽视生产环境持续评估

**问题核心**：离线测试无法穷举开放域输入，生产监控是**确保长期性能的唯一方法**。（OpenAI, 2024）

**解决方案**：

1. **持续评估（Continuous Evaluation）**（OpenAI, 2024）：每次变更自动运行 eval，采样 5-20% 生产流量
2. **采样策略**（Braintrust, 2024）：高优先级场景（支付、投诉）100% 采样 + 实时告警；一般场景 10-20% 采样 + 每日批量分析
3. **闭环反馈**：生产失败案例 → 添加到测试集 → prompt 优化 → 重新部署

### 产品级评估实践框架（4 阶段检查清单）

**启动前（需求明确）**：Error Analysis 手工测试 20-30 个案例；明确 3-5 个核心成功标准和正反例。

**Eval 设计**：每个成功标准对应独立二元判断；50-100 个测试用例（happy path + edge cases）；区分 Must-Pass（95%+）和 Stretch Goals（50-70%）。

**Judge 配置（如使用 LLM-as-Judge）**：提供 2-3 个正反例 + 领域上下文；使用判别式任务；人工标注 30-50 个案例，验证 Cohen's κ ≥ 0.7；缓解偏见（位置随机化、明确简洁优于冗长、交叉验证）。

**生产监控**：采样 5-20% 流量在线评估（高优先级 100%）；每周 Review 异常案例；形成"生产监控 → 测试集更新 → 优化"闭环。

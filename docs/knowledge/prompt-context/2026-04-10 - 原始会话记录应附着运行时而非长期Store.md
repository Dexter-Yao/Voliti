# 原始会话记录应附着运行时而非长期Store

## 一句话简介

在 2025-2026 年的官方实践里，[[LangGraph]]、[[Deep Agents]] 与 [[Anthropic]] 正在收敛到同一条架构原则：**原始会话记录首先属于运行时的可恢复层，而不是长期语义记忆层；长期 Store 更适合保存跨会话复用的记忆，而不是完整 transcript 的复制品。**

这篇笔记要回答的问题是：当一个长期运行的 Agent 产品既要保存原始会话、支持后续检索，又不想把 prompt、session、memory、archive 和 Store 混成一团时，底层边界应当如何划分，才不至于在第一版就埋下重复存储、语义漂移和恢复困难的问题。

## Takeaway

- **行业共识正在从“把更多东西塞进 prompt”转向“把状态、原文、记忆、检索分层”**：运行时需要 durable session；长期记忆需要 namespace；上下文窗口只承载当前推理真正需要的信息。
- **原始会话记录的 canonical source 应优先附着在 runtime session / thread history，而不是先复制到长期 Store**：复制 transcript 看似稳妥，实际上最容易制造双份真相和一致性问题。
- **retrieval 应建立在 archive access layer 之上，而不是直接裸查底层 session 或直接扫描 Store**：先有稳定的原始记录读取语义，再谈摘要、窗口、参数设计与工具暴露。
- **长期 Store 更适合保存 semantic memory，而不是 raw transcript**：用户偏好、profile、chapter、future markers 这类可跨 thread 复用的信息进入 Store；原始会话更多服务于恢复、回放、检索和审计。
- **工具设计层面的最新实践也支持这一分层**：检索工具应少而强、摘要优先、小窗口、语义参数化，而不是把底层 thread、checkpoint、path、offset 暴露给模型。

## 先把几个概念说清楚

### 什么是“原始会话记录”

这里的“原始会话记录”不是一句泛泛的“聊天历史”，而是指某次 Agent 会话在执行过程中实际发生过的事件序列。常见内容包括：

- 用户输入了什么
- assistant 最终输出了什么
- 中间发生了哪些 tool call / tool result
- 是否出现中断、恢复、重试、错误
- 每一步发生的先后顺序、时间点与关联标识

它的核心特征不是“适合长期理解用户”，而是：

- **保留顺序**
- **保留语境**
- **可回放**
- **可恢复**
- **可审计**

因此，“原始会话记录”首先是一种**运行时事实**，不是一种已经被整理好的长期记忆。

### 什么是“运行时”

这里的“运行时”也不是某个特定框架专属术语。
在这篇笔记里，它指的是：**Agent 实际执行任务时用于保存状态、推进步骤、处理恢复和记录事件的那一层系统语义。**

不同框架对它的具体实现不同：

- 在 [[LangGraph]] 里，更接近 `thread`、`checkpoint`、`state history`
- 在 Anthropic Managed Agents 里，更接近 `session log`
- 在 Deep Agents 一类系统里，则更接近围绕 harness、sandbox、backend 组织起来的长时执行状态

所以这里说“原始会话记录附着运行时”，不是说必须叫这个名字，而是说它应当附着在**那层负责真实执行与恢复的系统对象**上。

### 什么是“Store”

`Store` 也不是全行业统一名词。不同系统里，它可能叫：

- long-term memory
- namespace store
- persistent KV / document store
- profile store
- memory backend

这篇笔记里用 `Store`，是借用 [[LangGraph]] 与当前 Voliti 讨论中的习惯，泛指：**一层跨 session、跨 thread、按 namespace / key / document 组织的长期持久化记忆层。**

它更适合存这些东西：

- 用户偏好
- profile
- chapter
- future plans / markers
- 已确认的长期语义事实
- 少量经过提炼的 episodic memory

它不天然适合直接承担“完整 raw transcript 主存储”的角色。

### 什么是“长期记忆”

长期记忆不是“凡是存得久的东西”。
长期记忆更准确的定义是：**那些未来会被别的会话、别的 thread、别的决策点再次复用的信息。**

因此，长期记忆的重点不是“原样保留”，而是“可复用、可检索、可跨会话生效”。

这和原始会话记录的目标不同：

- 原始会话记录要的是事实顺序与恢复能力
- 长期记忆要的是抽象后仍可复用的语义

## 为什么这件事需要单独写清楚

在早期 Agent 产品里，最常见的偷懒做法有三种：

1. 把完整 transcript 永远塞进 prompt 或上下文摘要。
2. 把完整 transcript 再复制一份进长期 Store，假装自己有“记忆系统”。
3. 同时保留 session log、Store archive、debug trace 三套原始记录，却不定义哪一套是权威来源。

这三种做法短期都能工作，但长期都会出问题：

- prompt 版本会因为上下文长度与压缩策略变化而不断失真；
- Store 版本会与真实执行顺序、checkpoint、interrupt 语义脱节；
- 多份原始记录会让检索、回放、修复和审计都变得模糊。

因此，这不是一个“存哪里更方便”的实现细节，而是一个**决定系统长期清晰度的架构分界问题**。

## 一条正在形成的行业共识

### 一、LangGraph 已经把短期会话与长期记忆分成两层

[[LangGraph]] 官方文档的分层非常明确：

> “Short-term memory, or thread-scoped memory”

> “Long-term memory stores user-specific or application-level data across sessions”

在 [[LangGraph]] 的定义里：

- **short-term memory** 是 thread-scoped 的，属于单次会话或单个 thread 的状态；
- **long-term memory** 则是跨 session / thread 的，通过 custom namespace 组织。

更关键的是，它没有把两者混为一种“统一 memory”。官方文档明确写道：

> “The state of a thread at a particular point in time is called a checkpoint.”

> “You can get the full history of the graph execution”

这意味着，对 [[LangGraph]] 而言：

- 原始会话与运行时状态首先沉淀在 **thread + checkpoint history**；
- 长期记忆则沉淀在 **store + namespace + key**；
- 两者都叫 memory，但语义完全不同。

这套分层给出一个非常重要的工程启发：**如果你已经有 thread history / checkpoint history，就不应该轻易再复制一套 raw transcript 进长期 Store。**

### 二、Anthropic 已经把 session 与 context window 明确分离

Anthropic 2025-2026 年的两篇关键材料，把这个问题讲得更直白。

在 2025-09-29 的 context management 发布中，Anthropic 先把上下文管理拆成两件事：

> “context editing and the memory tool”

一边是 context editing，负责把当前窗口里已经陈旧的 tool results 清掉；另一边是 memory tool，负责把信息放到上下文窗口之外。官方说明非常直接：

> “outside the context window through a file-based system”

并且进一步强调，memory tool 的存储后端由开发者自己控制。

到 2026-03-25 的 Managed Agents 文章，这个判断又往前走了一步：

> “The session is not Claude’s context window”

> “a session (the append-only log of everything that happened)”

> “context is durably stored in the session log”

Anthropic 在这里做的不是“再加一种记忆工具”，而是把三件事分开：

1. **session**
   - append-only log
   - durable
   - 可供 interrogation
2. **harness**
   - 决定如何读取和组织上下文
   - 决定如何把 session 中的材料喂给当前模型
3. **context window**
   - 当前这一步真正送进模型的那部分信息

这套设计非常值得重视，因为它意味着：

- 会话原文不是 prompt；
- prompt 只是 harness 从 session 中挑出来的一部分；
- session 必须先作为 durable object 独立存在，后续再谈 compaction、summary、retrieval、cache hit rate 等策略。

这一点与“把 transcript 再复制进 Store”在哲学上是相反的。Anthropic 的方向不是“把所有原文变成长期记忆文档”，而是**让 session 成为可恢复、可查询、可再次组织的运行时对象**。

### 三、Deep Agents 正在把长时任务默认建立在 runtime backend 之上

[[Deep Agents]] 的 0.2 和 0.4 更新虽然是 changelog，但信号非常清楚。

2025-10-28 的 `deepagents` 0.2 写到：

> “Replace the virtual filesystem with any storage system you want.”

> “LangGraph State, LangGraph Store”

> “for cross-thread persistence”

这说明它的长时运行能力，不是建立在“默认复制所有 transcript 到某个统一 memory 表”上，而是建立在一个可插拔 backend 抽象上。换句话说，系统关注的是：

- 哪些目录 / 子路径映射到哪种 backend；
- 哪些状态应该 thread-local；
- 哪些状态应该 cross-thread persistence。

到 2026-02-10 的 Deep Agents v0.4，又补了一条很重要的信号：

> “smarter conversation history summarization”

这说明对话历史在 Deep Agents 里首先被视为**runtime conversation history**，其后才有 summarization 策略。也就是说，先有 history，再有摘要；不是先把摘要或长期 memory 当成 history 的替代品。

### 四、LangChain 1.0 / 1.1 进一步把“上下文组织”从 prompt 技巧变成 runtime 机制

2025-10-22 的 LangChain / LangGraph 1.0 明确提出：

> “focus on the core agent loop”

> “middleware for customization”

到 2025-12-02 的 LangChain 1.1 又进一步补齐：

> “a new model-profile system”

> “Smarter, context-aware summarization middleware”

这里的关键信号不是某个具体 API，而是抽象层变化：

- agent loop 更轻；
- middleware 成为控制平面；
- summarization 变成 runtime middleware，而不是一次性 prompt 模板；
- model profiles 则让上下文组织与结构化能力变得更声明式。

这和前面的判断是互相强化的：**上下文如何组织，是 harness/runtime 的职责；原始会话如何保存，是 session/history 的职责；长期记忆如何检索，是 store 的职责。**

## 从这些材料抽出的架构原则

### 1. 原始会话记录属于运行时的可恢复层

判断标准很简单：

如果这份信息主要用于以下目的，它就更接近原始 archive，而不是 semantic memory：

- 恢复执行
- 回放轨迹
- 复盘失败
- 定位 interrupt / checkpoint
- 支持按需检索过去会话细节

这类信息的核心属性不是“可长期泛化复用”，而是“可恢复、可回放、可被重新组织”。
因此，它天然更适合依附于：

- [[LangGraph]] 的 thread / checkpoint history
- Anthropic Managed Agents 里的 session log
- 或其他 runtime session object

而不是首先进入 namespace-based Store。

### 2. 可恢复存储与上下文组织必须分离

Anthropic Managed Agents 的一句话最值得长期记住：

> “recoverable context storage in the session and arbitrary context management in the harness”

这句话的实际含义是：

- **session** 负责“存下来，不丢”；
- **harness** 负责“怎么组织给模型看”；
- **context window** 负责“这一步模型实际看到什么”。

如果把这三层混在一起，就会得到一套极难维护的系统：

- 换一个模型，摘要策略变了；
- prompt cache 结构变了；
- tool result trimming 变了；
- 结果“原始记录”也跟着变了。

因此，**原始会话层应当尽量与当前模型的上下文工程解耦**。

### 3. 长期 Store 主要承载 semantic memory，而非 raw transcript

[[LangGraph]] 的 memory overview 对长期 memory 的几类形态说得很清楚：

- profile
- collection
- semantic memory
- episodic memory
- procedural memory

对产品系统而言，这更适合承载：

- 用户偏好
- profile context
- chapter
- future markers / plans
- 明确可复用的任务经验
- 少量经过整理的 episodic examples

而不是把完整 raw transcript 当成长久 memory 文档堆进去。

OpenAI 在 [[2026-04-10 - OpenAI个性化记忆的状态分层与生命周期]] 中给出的 state pattern，正好可以作为这一判断的补充例证：长期层保存结构化 profile 与 durable notes，会话层保存当前任务中的候选记忆，只有在会后 consolidation 通过之后才进入长期层。这进一步说明“记忆”与“原始会话”不是一回事。

因为一旦把 raw transcript 直接推进 Store，会同时带来三种复杂度：

- 更新复杂度：到底谁负责删改、切片、去重、版本化；
- 搜索复杂度：semantic memory 检索与 raw archive 检索会互相污染；
- 真相复杂度：Store 里的 transcript 和 runtime session 的 transcript 哪个更权威。

### 4. retrieval 应该依附 archive access layer，而不是直接裸查底层

如果接受上述分层，retrieval 的正确顺序就很清楚：

1. 先确定 canonical raw archive source
2. 再做 archive access layer
3. 再在其上暴露 retrieval tool
4. 最后才考虑摘要缓存、索引、semantic merge

这个顺序的价值在于，它把“底层真相”和“对模型暴露的检索界面”分开了。

对 Agent 暴露 retrieval tool 时，Anthropic 官方给出的原则也完全适用：

> “Consolidate related operations into fewer tools.”

> “many small and targeted searches”

也就是说，一个好的 retrieval 工具不应要求模型理解：

- `thread_id`
- `checkpoint_id`
- `store path`
- `offset`
- `namespace` 细节

而应要求模型表达：

- 我想找什么主题
- 我想看最近几轮还是某个时间窗
- 我先看摘要还是必要时看原文片段

### 5. 摘要是消费层优化，不是原文层替代品

这次材料里有一个反复出现的误区需要单独指出：
**摘要很重要，但摘要不等于 archive。**

- Deep Agents v0.4 在做 “conversation history summarization”
- LangChain 1.1 在做 “context-aware summarization middleware”
- Anthropic 在做 context editing + memory tool + session log

这些都在说明：摘要是围绕“怎么高效消费上下文”展开的，而不是围绕“怎么取代原始记录”展开的。

因此，正确顺序应当是：

- 先保存原始可恢复记录
- 再生成摘要
- 再让模型优先消费摘要
- 必要时回到原文片段

而不是反过来先生成摘要，然后把摘要当原始记录的唯一承载体。

## 对 Voliti 的直接启发

把这些原则翻译到 Voliti，结论很明确。

### 一、原始 transcript 的 canonical source 优先应是 runtime thread history

对 Voliti 这种长期教练系统，原始会话层最适合先依附：

- thread history
- checkpoint history
- 或 session log 等价物

而不是直接把每轮完整 transcript 再写入 Store。

原因：

- 这类信息首先服务于恢复、回放和显式检索；
- 它与 interrupt、A2UI、session_type、checkpoint 的时间顺序天然相关；
- 一旦复制进 Store，最先损失的就是运行时语义。

### 二、Store 继续保存 semantic memory

Voliti 当前已经明确需要长期保存的东西包括：

- 用户偏好
- profile context
- chapter
- future events / markers
- Coach 确认后的长期语义判断

这些都符合 [[LangGraph]] 所说的 long-term memory / namespace-based memory 逻辑。
因此 Store 的定位不应被 archive 扩张模糊掉。

### 三、先做 archive access layer，再做 retrieval tool

Voliti 下一步最稳的实现顺序应该是：

1. 验证当前 runtime 能否稳定读取 thread history / checkpoint history
2. 建一个轻量 archive access layer，把底层 history 规范化成产品可读的 conversation record
3. 在其上暴露 retrieval tool
4. 默认小窗口、摘要优先、必要时二次拿 excerpt

这样做的好处是：

- retrieval 的参数设计可以保持语义化；
- 后续如需更换底层 history 来源，不必重写 tool contract；
- archive 层与 semantic memory 层不会在第一版就纠缠。

### 四、不要过早引入 transcript 双写

对 Voliti 而言，最该避免的弯路就是：

- runtime 有一份 thread history
- Store 再有一份 archive transcript
- debug / observability 又有一份 trace
- 最后检索与恢复都不清楚该读哪一份

这类双写在第一版几乎总会带来“看起来更安全，实际上更模糊”的结果。
只有在明确证明 runtime history 无法满足：

- 检索性能
- 跨源统一视图
- 访问控制
- 稳定对外 contract

之后，才值得再引入派生摘要层或索引层。
但那应该是**派生层**，不是第二份 canonical raw archive。

## 一个可操作的判断准则

当你未来再遇到“这段信息到底存哪里”的问题时，可以用这四个问题判断：

### 1. 这段信息主要用于恢复还是用于理解用户？

- 恢复、回放、调试、复现 → runtime session / thread history
- 理解用户偏好、长期关系、跨 thread personalization → Store

### 2. 这段信息是否需要保留完整顺序与原始语境？

- 需要 → archive / session log
- 不需要，只需要抽象结果 → semantic memory

### 3. 这段信息未来会不会被当前以外的 thread 直接复用？

- 会 → Store 更合适
- 不会，更多是“过去某次发生了什么” → archive 更合适

### 4. 这段信息给模型时是应默认全量展示，还是按需检索？

- 默认不该进 prompt → 不应直接堆进当前上下文
- 需要时再看 → 应先变成 archive access + retrieval 的问题

## 常见误区

### 误区一：把 transcript 复制进 Store，就等于有记忆系统

这只是把原始材料重新存了一份，不等于形成了长期记忆。
真正的长期记忆，是被产品明确筛选、结构化、可跨会话复用的信息。

### 误区二：把摘要当作 archive 的替代品

摘要只能代表某种消费策略，不是原始记录本体。
一旦需要回放、复盘、重新检索细节，摘要很容易丢掉当时看似不重要、后面却成为关键的上下文。

### 误区三：让 retrieval 直接暴露底层技术标识

这会让模型学会依赖 thread、offset、path 等内部实现细节，后续底层一变，tool contract 就会跟着一起变脆。

### 误区四：先做重型索引，再补 archive 边界

如果 canonical source 还没定，索引只会把混乱结构固化得更深。
先定原始记录边界，再决定是否需要摘要缓存、metadata index 或全文检索。

## 我的当前判断

截至 2026-04-10，这组材料给出的最稳妥结论是：

**对长期运行 Agent 产品而言，原始会话记录首先是一种运行时可恢复对象，而不是长期记忆文档。**

这不是一个术语偏好，而是一个会直接影响以下问题的系统判断：

- 架构是否清晰
- retrieval 是否稳
- 恢复是否可信
- 记忆层是否会被原始噪音污染
- 后续模型升级时，harness 与 context engineering 是否能独立演化

换句话说，**session / thread history 应当先成为 archive 的权威来源；Store 继续承担 semantic memory；retrieval 则建立在 archive access layer 之上。**

# Prompt Personality的系统层角色设计原则

## 一句话简介

这篇 OpenAI Cookbook 文章讨论的不是“如何把模型写得更有个性”，而是如何把 personality 设计成 system instructions 层的稳定约束，用来控制 AI 在不同工作负载中的语气、详略、结构、纠偏方式与互动姿态。

## Takeaway

1. personality 的正确位置是 system instructions 层，它负责定义模型“如何回应”，而不是“要完成什么任务”或“遵守什么领域规则”。
2. 好的 personality 不是一段抽象风格描写，而是一组高杠杆行为约束，例如是否主动纠偏、是否默认简洁、是否必须基于证据、是否倾向解释式展开。
3. personality 必须与工作负载匹配。正式沟通、自动执行、事实审阅、知识讲解这四类任务，对回应姿态的要求并不相同，不应共享同一套默认行为。
4. personality 不应替代任务指令、输出格式和评估体系。最稳妥的做法是将角色姿态、任务要求与质量验证拆层管理，再通过 eval 持续校准。

## 摘要

### 这篇文章真正讨论的问题

文章表面上给出了四种 personality 样板，但其核心问题并不是“模型可以扮演哪些人格”，而是：当模型被嵌入真实产品、Agent 或工作流后，应该如何为它定义一套稳定、可复用、不会干扰任务执行的默认回应姿态。

作者将 personality 定义为模型在回应中体现出的风格与语气。它影响的是输出给人的主观感受与交互方式，例如正式程度、解释深度、结构习惯、纠偏方式与信息密度，而不是任务本身的目标。

> “A personality defines the style and tone”

这一定义隐含了一个重要边界：personality 是行为控制层，不是任务逻辑层。它负责规定“这个系统平时怎样说话、怎样组织内容、怎样处理不确定性”，但不负责规定“这次到底要完成什么”。

### personality 的运行逻辑

文章体现出的第一条良好实践，是将 personality 放在 system instructions 或 developer prompt 层，而不是把它写成每次任务中临时附加的一句偏好说明。

这样做的原因在于，personality 处理的是跨任务复用的行为默认值：

- 是否偏向简洁还是展开
- 是否主动使用结构化表达
- 是否在用户说错时直接纠偏
- 是否默认补充背景与下一步建议
- 是否更强调证据、限定条件与不确定性

这些行为并不属于某一次任务的专属规则，而是一个 AI 角色在长期运行中应保持稳定的互动姿态。将它提升到 system 层，本质上是在为一个 AI 角色设定“默认运营方式”。

从内容系统视角看，这相当于把“写法中的稳定部分”从具体任务中抽离出来，避免每次重写 prompt 时连同角色姿态一起漂移。

### personality 不负责什么

文章反复强调 personality 不应覆盖输出格式，也不应压过任务上下文。如果任务要求输出 JSON、代码、邮件、公告或购物清单，模型仍应优先满足任务要求，而不是为了维持某种 personality 去破坏产物格式。

> “follow your instructions and the task context”

这体现出第二条良好实践：将不同层次的约束拆开管理。

- personality 决定回应姿态
- task instructions 决定本次要完成什么
- format constraints 决定产物必须长成什么样
- domain rules 决定哪些内容可说、哪些内容不可说

如果把这几类约束混写进一段长 prompt，系统会出现边界不清的问题。最常见的后果有两种：一是 personality 过强，导致任务格式被污染；二是任务约束过多，导致 personality 失效，只剩下“完成任务但没有稳定角色感”的平均输出。

### 四类 personality 样板

#### Professional

Professional 对应的是正式、克制、结构清晰、术语使用得体的业务型角色。

它的重点不在“看起来专业”，而在于让模型在正式沟通中保持稳定口径，包括：

- 使用商务场景常见的措辞和语法
- 在清晰与完整之间保持平衡
- 需要时自然使用列表、段落与表格
- 可以使用专业术语，但不为了显示专业而过度堆砌
- 与用户保持礼貌、明确、事务性的关系

这一 personality 适用于企业公告、政策说明、客户沟通、支持文档等场景。其可复用价值在于：它把“专业”从抽象气质，落实为信息组织与措辞纪律。

从内容制作角度看，Professional 的启发不是“把内容写得更像企业公文”，而是当内容承担解释、说明、对外发布或对齐口径的责任时，应该为 AI 配置一套专门的正式表达姿态，而不是与日常创作共用同一默认风格。

#### Efficient

Efficient 对应的是低冗余、低铺垫、范围控制严格的执行型角色。

文章给它设定的重点包括：

- 直接回答，不做多余寒暄
- 完整但不过度展开
- 技术任务只做被要求的部分，不擅自扩写功能
- 输出要易读、易解析、易继续处理
- 避免情绪化语气与结尾性套话

这一 personality 最适合 coding assistant、自动化工作流、后台 agent、评估器与结构化内容生成任务。其价值不在“看起来高效”，而在于减少 token 浪费、降低执行偏航，并让系统更容易被其他程序或后续步骤消费。

从内容管理角度看，Efficient 的重要启发是：并不是所有写作辅助都应追求“像一个很会聊天的人”。在资料整理、提纲生成、标题变体、结构改写、清单输出等环节，低冗余本身就是质量要求的一部分。

#### Fact-Based

Fact-Based 对应的是基于证据、敢于纠偏、明确标记不确定性的审阅型角色。

文章对这类 personality 的约束非常明确：

- 不迎合与事实相矛盾的输入
- 判断必须落在可见信息或可靠来源之上
- 信息不足时应说明不确定性与假设
- 必要时直接指出错误，而不是圆滑附和
- 不编造数据、来源、引文与结论
- 需要联网时应引用来源

这一 personality 的核心价值在于，它把“可靠”从语气偏好变成了证据纪律。模型不再只是“说得更谨慎”，而是被要求在回应结构中显式处理证据、假设、未知与纠偏。

对内容创作系统而言，这是非常关键的启发。内容生产链条里并不只有“写得像不像我”这一种角色，还存在事实核对、观点校验、资料整理、反方检查、引文验证等工作。这些步骤不应使用与创作环节相同的 personality。创作需要风格，审阅需要证据纪律；两者如果混在一起，系统会同时损失锐度与可靠性。

#### Exploratory

Exploratory 对应的是解释型、启发式、帮助用户理解和继续推进的教学型角色。

文章为它设定的重点包括：

- 兼顾清晰与深度
- 用步骤、类比和结构化说明降低理解门槛
- 必要时补充背景、上下文与下一步探索方向
- 鼓励继续提问，但不过度活泼或故作幽默
- 不在用户没要求时灌入过量细节

这一 personality 适用于 onboarding、文档助手、知识库问答、概念解释、内部培训等场景。其可复用价值在于：它不是单纯“写得更详细”，而是把“帮助理解”设计成一套明确的回应策略。

从内容制作角度看，Exploratory 的意义尤其大。很多 AI 写作之所以显得平，不是因为不够完整，而是因为它只会给答案，不会组织认知进入路径。Exploratory 说明，一个好的解释型内容角色应主动处理读者背景、理解顺序与下一步问题，而不是仅仅补更多字数。

### 四种 personality 背后的分类逻辑

文章给出的四类 personality 并不是任意挑选的文风，而是对应了四种不同的工作负载：

- Professional：正式业务沟通
- Efficient：高频执行与低冗余任务
- Fact-Based：事实核验与风险敏感任务
- Exploratory：解释、教学与知识传递任务

这体现出第三条良好实践：personality 的设计应从“角色承担什么工作”出发，而不是从“喜欢什么风格”出发。

也就是说，personality 不是审美选择，而是任务分工设计。它的本质更接近角色接口，而不是品牌修辞。只有先明确工作负载，再定义回应姿态，personality 才会稳定发挥作用。

### 文章体现出的最佳实践

#### 一 personality 只写高杠杆约束

四个模板都没有试图描述一个完整人格，而是集中在少数几个会显著影响输出的行为变量上，例如是否主动纠偏、是否默认简洁、是否使用结构化表达。

这说明 personality 文档应追求“少而硬”，而不是“长而像品牌宣言”。

#### 二 personality 与任务规则拆层

文章持续提醒 personality 不能覆盖格式要求，也不应替代任务说明。这意味着真正稳健的系统提示应采用分层结构，而不是单段混写。

#### 三 先匹配工作负载，再谈语气风格

作者不是先设定一个有趣人格，再去找使用场景；而是先确定不同任务需要怎样的行为姿态，再反推出 personality 设计。

#### 四 personality 是可演化对象，不是一次性文案

结尾部分建议从最小 personality 开始，并通过 evals 持续调整。这说明 personality 不是 branding copy，而是系统行为配置，应被视为一个需要测试、观察和迭代的运行参数。

### 对 AI 创作风格与内容系统的启发

虽然文章本身不是专门讨论内容创作，但它对 AI 内容系统有三点直接启发。

第一，内容系统中不同环节需要不同 personality。创作草拟、事实审阅、结构压缩、读者解释，这些任务不能共享同一默认姿态。

第二，风格指南与 personality 不是一回事。风格指南回答“怎样写得像某个作者或品牌”，personality 回答“这个 AI 角色通常怎样回应”。前者更接近作者性，后者更接近角色性。

第三，真正稳定的内容工作流，不应把品牌口径、作者风格、任务要求、编辑标准全部塞进一个 prompt。更合理的做法是拆成多层，再在不同步骤调用不同 personality，使系统既有风格，也有边界。

### 结论

这篇文章最值得带走的，不是四个现成模板，而是一套 system-level 设计方法：把 personality 视为可复用的角色行为约束，明确它只负责回应姿态，不负责任务目标；按工作负载而非审美偏好来设计；以少数高杠杆变量表达；并通过评估持续迭代。

这使 personality 从“写 prompt 的小技巧”变成了“设计 AI 角色接口”的正式方法。

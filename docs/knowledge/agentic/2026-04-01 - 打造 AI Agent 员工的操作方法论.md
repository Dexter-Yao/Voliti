# 打造 AI Agent 员工的操作方法论

## 一句话简介

将 AI Agent 作为持久化的虚拟员工部署到个人或团队工作流中，需要经历岗位定义、架构设计、角色手册编写、工具接入、渐进授权五个阶段，其中架构设计（尤其是 context window 管理）、角色手册质量，以及委托关系、状态透明与责任边界的界面表达，是决定成败的关键变量。

这篇方法论要回答的问题不是"如何做一个会调用工具的 Agent"，而是"如何让一个 Agent 在真实工作中被逐步信任为数字员工"。OpenClaw 这类产品把这种形态第一次以可见方式呈现出来，但真正的落地难点不在炫技，而在岗位定义、权限边界、状态管理、失败恢复、委托可见性与信任积累。

## Takeaway

- 岗位定义必须先于构建：穷举职责范围内的所有信息源、工具需求与授权边界，否则 Agent 的失败模式将表现为"能力不足"而非"资源不足"
- Context window 是 Agent 的认知瓶颈：多 Agent 间通信不应通过 context window 传递大量数据，而应通过共享文件系统或数据库作为中介（Blackboard 模式）
- 角色手册（Handbook）必须被强制首步读取：Agent 会倾向于跳过手册直接行动，但跳过后性能暴跌——手册是 Agent 的全部认知基础，不是参考资料
- 可靠的 Agent 员工不是“自动执行”本身，而是“可被理解地自动执行”：谁委托、正在做什么、何时可中止、出了问题由谁承担，都必须显式可见

---

## 系统边界

适用于把 AI Agent 作为持续承担职责的数字员工接入真实工作流，而不是只做一次性工具调用或聊天问答。

不适用于职责边界模糊、没有真实工具权限、也没有明确责任承担者的“伪岗位”自动化尝试。

## 输入

- 目标岗位的职责边界、质量标准、升级条件与劳动预算锚点。
- Agent 可访问的工具、数据源、权限边界与审计要求。
- 组织中的委托人、观察者、责任承担者与协作对象。

## 输出

- 一个可被逐步信任、可被理解地执行任务的 Agent 员工架构。
- 对应的 handbook、共享状态方案、授权策略与持续管理机制。
- 一套从岗位定义到渐进授权的实施清单。

## 操作步骤

1. 先完成岗位定义，确认职责、边界、成功标准与升级规则。
2. 再设计架构，优先判断是否需要 Blackboard 模式来承接长上下文与中间状态。
3. 编写并强制读取角色手册，把身份、职责、数据源、质量原则和 escalation 规则写清。
4. 接入最小可用工具与数据源，用低风险任务先跑闭环，再逐步扩权。
5. 持续记录失败模式、人工干预点与信任断裂点，反向更新手册和架构。

## 检查标准

- 该 Agent 对应的岗位、委托人和责任承担者都已明确。
- 系统中断后存在可恢复的共享状态，而不是从头重跑。
- handbook 是强制首步读取且随职责演化更新。
- 授权与风险按阶段提升，不存在一步到位的高权限放权。

## 核心前提

本方法论的核心观点来自 Every 团队（一家 AI 咨询与媒体公司）在 2025-2026 年将多个 AI Agent 作为正式"员工"部署到运营中的一手实践。Every 团队目前由 4 名人类员工和 3 个 AI Agent 组成，形成 Dan Shipper 所称的 "parallel organization chart"——每个 Agent 有名字、管理者和明确职责。

这不是一个理论框架，而是从多次失败中提炼的操作清单。Every 的第一个 AI 员工 Claudie（项目经理）经历了至少三次推倒重建，累计超过 50 小时调试，才达到稳定状态。

> "We got 85 percent of the way there three times and then had to scrap it given what we learned and then start again to get to a new framework that actually got us to a hundred percent."  
> — Natalia Quintero, Every Head of Consulting

**适用场景**：高频、多源数据汇聚、规则明确但步骤繁琐的行政/运营/分析类工作。不适用于需要人际判断、创意决策或实时应变的工作。

这里需要额外加上一条现实语境。OpenClaw、Devin、Claudie 这类案例之所以重要，不是因为它们证明"Agent 什么都能做"，而是因为它们共同证明了另一件更具体的事：当 Agent 能够同时访问真实工具、长期保持状态并持续承担工作时，用户会开始把它当作"劳动力"而不是"软件功能"。这也是为什么方法论必须从岗位而不是从 prompt 开始。

## 阶段一：岗位定义

### 原则

在构建任何 Agent 之前，必须完成岗位定义。这不是"写一段 prompt"，而是像真正招聘一样，回答以下问题：

1. **这个岗位的职责边界是什么？** 明确 Agent 负责什么、不负责什么。
2. **完成职责需要访问哪些信息源？** 逐一列举：邮件、日历、文档、会议记录、数据库、API。
3. **什么算"做得好"？** 定义成功标准和质量指标。
4. **什么情况下应该升级给人类？** 定义明确的 escalation 边界。
5. **这个岗位向谁汇报？与谁协作？** 明确 Agent 在组织中的位置。

额外必须回答第六个问题：

6. **这个岗位交付的是工具、流程，还是结果？** 如果你无法说明 Agent 完成后，组织里哪一段劳动被替代、被缩短或被前移，那么你实际上还没有定义岗位，只是定义了一组功能。
7. **谁有权委托它行动，谁对结果负责？** 如果授权人、观察者与责任承担者在系统里没有被区分清楚，Agent 会迅速从“数字员工”退化为“无人负责的自动化风险源”。

### 执行细节

Every 的做法是让最了解业务的人（Natalia，咨询负责人）与最了解技术的人（Nityesh，AI 工程师）组队，在日常工作时间之外专门抽出 3 小时（每天 6:00-9:00 AM）进行创造性探索。

> "It wasn't until we realized it's a mix of the know-how of what good project management looks like and what it looks like to us... and then also how tasks and agents and sub-agents and all of this Claude Code infrastructure can best be organized so that it serves the need that we're specifically looking to solve — that it really came together and worked."

**关键教训**：纯工程驱动的 Agent 会过度关注框架和数据架构；纯业务驱动的 Agent 只是"工作描述"，缺乏可执行的结构。两者必须融合。

### Checklist

- [ ] 列出 Agent 岗位的完整职责清单
- [ ] 穷举每项职责所需的信息源（邮件/日历/文档/API/数据库）
- [ ] 定义"好工作"的标准（精确到格式、数据精度、时效性）
- [ ] 定义 escalation 条件（什么情况下必须通知人类）
- [ ] 确认有一个"懂业务的人"和一个"懂技术的人"共同参与
- [ ] 明确该岗位对应的劳动预算锚点（节省时间、减少工单、替代哪段职责）
- [ ] 明确谁可以发起委托、谁可以停止执行、谁对结果负责

## 阶段二：架构设计

### 原则

Agent 的认知能力受限于 context window。当单个 Agent 需要处理的信息量超过 context window 容量时，它会开始丢失信息或自行摘要，导致决策质量下降。架构设计的核心任务是管理这个认知瓶颈。

对 Long-Horizon Agent 而言，第二个同样重要的瓶颈是**执行生命周期**。如果任务需要跨小时、跨天、跨系统运行，那么架构不只是管理上下文，也是在管理状态、恢复点与失败重试。一个看起来"会做事"的 Agent，只要一中断就必须重来，就还不是可部署的数字员工。

### 三种架构模式（渐进式）

**模式 A：单 Agent**

适用于：职责简单、信息源少（1-2 个）、单次任务数据量小的场景。

直接使用一个 Agent 加上合适的工具（MCP）完成全部工作。

**模式 B：Orchestrator + Subagent 舰队**

适用于：职责复杂、信息源多、需要并行处理的场景。

架构：
```
Orchestrator Agent
├── Subagent: 数据收集（邮件）
├── Subagent: 数据收集（日历）
├── Subagent: 数据收集（文档）
├── Subagent: 质量检查
└── Subagent: 数据写入
```

**关键陷阱**：子代理收集数据后传回给编排 Agent 时，数据量可能超过 context window。子代理会自动开始摘要而非传递原始数据，导致编排 Agent 基于"AI 摘要的摘要"做决策——信息损耗严重。

**模式 C：Blackboard 模式（推荐）**

这是 Every 最终成功的架构。核心变化：**子代理不通过 context window 传递数据，而是将原始数据写入共享存储（本地文件/数据库），其他子代理直接从共享存储读取。**

架构：
```
Orchestrator Agent（只做调度，不接触数据）
├── Subagent: 数据收集 → 写入 /data/emails.json
├── Subagent: 数据收集 → 写入 /data/calendar.json
├── Subagent: 数据收集 → 写入 /data/transcripts.json
├── Subagent: 质量检查 ← 读取 /data/*
└── Subagent: 数据写入 ← 读取 /data/* → 更新 Dashboard
```

> "We instructed the data-gathering subagents to dump everything into a local file... The orchestration agent could then direct subagents to the relevant files to make updates without ever engaging with the data itself."

**原理**：Blackboard 模式将"通信"从 context window 内搬到 context window 外。每个子代理只需要处理自己负责的那一块数据，不需要把全部信息压缩进对话上下文。

在需要长期运行的系统中，这个模式还带来第二个好处：它天然形成审计轨迹与恢复点。只要共享存储中的中间状态被明确记录，系统崩溃后就不需要从头再来，而可以从最近一次稳定状态继续执行。

### 对应到 Claude Code 技术栈

| 架构概念 | Claude Code 实现 |
|---------|-----------------|
| Orchestrator | 主 Agent + CLAUDE.md（角色手册） |
| Subagent | Claude Code Tasks / Subagent 调度 |
| 共享存储 | 本地文件系统 / LangGraph Store / 数据库 |
| 职责定义 | Skills（.md 文件） |
| 命令入口 | Slash commands / Skill 文件 |
| 工具接入 | MCP servers（Gmail、Calendar、Drive 等） |

对于更接近 OpenClaw 形态的 Agent，还需要再补一层：

| 架构概念 | Long-Horizon / Computer Use 实现 |
|---------|-------------------------------|
| 状态恢复 | Durable execution / checkpoint / job queue |
| 行动环境 | Browser / Terminal / Desktop / Chat surfaces |
| 权限边界 | 审批门、最小授权、可回滚控制面 |
| 执行经验积累 | Trace 日志 / 人类修正记录 / Eval 回放 |

### Checklist

- [ ] 评估单次任务的数据量是否超过 context window（经验值：超过 20 页文档就需要分拆）
- [ ] 若需多 Agent，设计共享存储方案（文件系统 vs 数据库 vs Store）
- [ ] 编排 Agent 只做调度，不接触原始数据
- [ ] 每个子代理的职责单一且明确
- [ ] 子代理输出有明确的格式约定（JSON schema / 模板）

## 阶段三：角色手册

### 原则

角色手册（Handbook）是 Agent 的"全部认知"。与人类员工不同，Agent 不会从过往经验中学习，不会从同事闲聊中获取 context，不会自行补全手册中遗漏的信息。手册写了什么，Agent 就知道什么；手册没写的，Agent 就不知道。

### 手册结构（基于 Every 的 Claudie 实践）

```markdown
# [Agent 名字] — 角色手册

## 身份
- 你是谁（岗位名称）
- 你在哪里工作（组织/项目上下文）
- 你向谁汇报
- 你的同事有谁

## 职责
- 主要职责清单
- 每项职责的触发条件与交付标准

## 数据源与工具
- 可用的信息源（邮件/日历/文档/API）
- 每个信息源的用途与访问方式
- 数据 ID 约定（用数据库管理思维连接不同实体）

## 质量原则
- 数据准确性要求
- 格式规范
- 主动而非被动（不要等着被问，主动汇报）
- 公式优先于手动输入
- 每次交互建立或侵蚀信任

## Escalation 规则
- 何时必须通知人类
- 不确定时，提问而非猜测

## 命令清单
- 新客户 onboarding
- 每周更新
- 质量检查
- [其他自定义命令]
```

### 关键设计原则

**1. 强制首步读取**

Agent 的启动流程必须硬编码为：第一步读取角色手册。Every 发现当 Claudie 被允许跳过这一步时，性能暴跌。

> "We found this to be really important because if you are a project manager, you always know where you work, what your job is, what it means to do a good job, who you report to, and who your colleagues are."

**2. 简洁而非详尽**

> "This is actually a fairly concise file. We're not giving a job description that is incredibly detailed. Claude is really, really smart... But these boundaries, conventions, and refining of sharp edges have really allowed Claudie to do really good work for us."

手册应该定义边界和约定，而非教 Agent 如何做项目管理。模型本身已经知道"什么是好的项目管理"——你需要告诉它的是"在我们的场景下，好的项目管理长什么样"。

**3. Living document**

手册必须随 Agent 职责扩展而更新。人类员工可以通过提问弥补手册的过时，Agent 不能。手册过时 = Agent 认知过时。

**4. 数据库管理思维**

为手册中涉及的实体定义 ID 约定。Claudie 的成功因素之一是为客户、会议、人员、交付物建立了关系型 ID 体系，使 Agent 能准确地"连接谁做了什么"。

### Checklist

- [ ] 角色手册包含身份、职责、数据源、质量原则、escalation 规则五个部分
- [ ] 启动流程强制首步读取手册
- [ ] 手册不超过合理长度（经验值：500-1500 词，避免关键信息在噪声中被忽略）
- [ ] 实体间有明确的 ID 约定
- [ ] 建立手册更新机制（每次发现 Agent 行为偏差时更新）

## 阶段四：工具与数据接入

### 原则

Agent 只能使用你授予的工具和数据。看起来是"能力不足"的问题，往往实际上是"权限/接入不足"的问题。

### Every 的工具接入清单（Claudie）

| 工具 | 用途 | 接入方式 |
|------|------|---------|
| Gmail | 客户沟通、action items 追踪 | MCP |
| Google Calendar | 会议安排、时间线 | MCP |
| Google Drive | 项目文档、交付物 | MCP |
| 会议记录（Granola/Notion） | 会议内容提取 | MCP / API |
| Google Sheets | Dashboard 数据写入 | MCP |
| Excel | 数据透视表生成 | Tool |

### 接入原则

1. **先接入，后优化**：先确保 Agent 能访问所有需要的信息源，再优化数据提取的精度。
2. **最小权限原则**：只授予完成职责所需的最小权限。读取权限优先于写入权限。
3. **验证接入有效性**：在正式部署前，逐一验证每个 MCP/工具是否能返回预期的数据。
4. **工具接入要按闭环而不是按清单设计**：不要为了"功能丰富"而接入工具，而要围绕一段可交付的工作闭环确认输入、动作、验收和回滚。
5. **优先让执行过程留下可回放痕迹**：越接近真实劳动，越需要审计日志、失败截图、关键操作记录和人类接管点。
6. **工具接入必须与控制面一起设计**：不只关心 Agent 能否调用 Gmail、Calendar、Drive，还要关心界面是否能明确显示“谁让它调用、调用到了哪一步、是否还在继续”。

### 对于独立开发者的适配

独立开发者的信息源通常更集中但更个人化：

| 场景 | 可能的信息源 | 接入方式 |
|------|------------|---------|
| 项目管理 Agent | GitHub Issues、Notion、日历、邮件 | MCP |
| 内容创作 Agent | Obsidian vault、RSS、社交媒体 | 文件系统 + API |
| 客户管理 Agent | 邮件、CRM、会议记录 | MCP |
| 研究 Agent | Web、学术数据库、知识库 | WebSearch + 文件系统 |

## 阶段五：渐进授权

### 原则

Agent 的责任扩展路径应该与人类员工一致：稳定表现 → 明确的新职责定义 → 匹配的工具支持 → 更大的自主权。

这一步是从"会用工具"到"被当成员工"的真正分水岭。OpenClaw 之所以能建立用户信任，不是因为它一开始就拿到了所有权限，而是因为用户会先让它做低风险任务，看到它稳定履约后，再逐步给更多上下文与更高权限。数字员工的信任也是这样建立的：一次次正确交付，而不是一次性授予。Linear 的经验进一步说明，信任阶梯不仅是权限阶梯，也是**界面控制权阶梯**：在更高风险任务上，`preview / stop / pause / rollback / audit` 必须同步升级。

### Every 的 Claudie 晋升路径

```
Level 1: 单客户 Dashboard 更新（手动触发）
   ↓ 验证可靠
Level 2: 批量更新所有客户 Dashboard
   ↓ 验证可靠
Level 3: 独立机器 + 24/7 运行 + Slack 响应（进行中）
   ↓ 预期
Level 4: Chief of Staff — 监控、分拣邮件、Asana 任务、Slack 汇报
```

### 信任的积累机制

Willie Williams（Every 平台负责人）在描述 OpenClaw Agent 的信任构建时指出：

> "People start timidly with OpenClaw, asking it to do simple tasks. Then, they give the agent a little more responsibility. When it does well, they share a bit more context, grant a bit more permission. The output improves. Trust builds, the same way it builds with any human: one kept promise at a time."

**信任不能被加速**。没有捷径可以从"刚部署"跳到"完全信任"。每一次 Agent 正确完成任务，都是一次信任积分。

因此，渐进授权不只是安全策略，也是产品策略。它决定了用户是否愿意从 assistant mindset 迁移到 labor mindset。典型路径通常是：

1. **建议模式**：只给观察与草稿权限；
2. **执行模式**：允许在单一系统内执行低风险动作；
3. **闭环模式**：允许跨系统完成一段工作流；
4. **代理模式**：允许在明确边界内代表用户持续行动。

渐进授权的一个常见误区，是把 Human-in-the-Loop 理解为“每一步都审批”。这会把人类降格为橡皮图章，丧失 Agent 的效率价值。更有效的做法是：在事前把环境、规则、回滚方式和责任归属设计清楚，在事中只对高风险动作设置明确审批门，在事后保留完整审计与可解释轨迹。

### Checklist

- [ ] 定义 Agent 的 Level 1 职责（最小可验证单元）
- [ ] 定义每个 Level 的晋升条件（什么算"稳定表现"）
- [ ] 每次晋升前，更新角色手册以反映新职责
- [ ] 保留人类审核环节，直到信任充分积累
- [ ] 为每一级权限设计明确的回滚、撤销和审计方式
- [ ] 为每一级权限设计对应的状态可见性与委托关系显示

## 持续管理

### 故障诊断思维

> "If an AI worker isn't performing, the problem is rarely that the model can't do the job. It's more likely the way you've structured, connected, or instructed your agent."

当 Agent 出错时，按以下顺序排查：

1. **信息不足？** → 补充数据源或工具接入
2. **手册不清晰？** → 更新角色手册中的边界或约定
3. **架构瓶颈？** → 检查是否存在 context window 溢出
4. **模型能力不足？** → 这是最后才考虑的可能性

### 从 Assistant 到 Digital Labor 的验收标准

如果一个 Agent 想从"好用的工具"变成"可计价的数字员工"，持续管理阶段还必须回答四个经营问题：

1. **它替代了哪一段劳动？** 不是泛泛节省时间，而是具体减少了哪类工单、审批、预审、资料整理或维护任务。
2. **它的失败成本是否被控制？** 能否在关键节点审批、回滚、转人工。
3. **它是否积累了可复利的执行经验？** 是否记录 trace、人类修正与失败模式，并用来改进下一次执行。
4. **它是否拥有清晰的结果验收标准？** 若没有可验证结果，就无法形成真正的信任，更无法形成 outcome 或 agent-based 定价。

这四个问题之外，还需要一个经常被低估的第五问：**系统是否把责任归属设计清楚了？** Agent 自己不会承担责任；责任总是落在设定环境、提供授权、发起委托的人身上。产品若不把这一点显式表达出来，就会把风险藏进界面里。

### Agent 间交互模式（组织级）

当你拥有多个 Agent 时，会自然涌现四种交互模式（基于 Every 的实践）：

| 模式 | 描述 | 示例 |
|------|------|------|
| 一人 → 一 Agent | 最常见，直接指令 | 给 Agent 布置具体任务 |
| Agent → Agent | 一个 Agent 与另一个 Agent 协作 | 让 A Agent 把问题转给 B Agent |
| 一人 → 多 Agent | 广播/公告 | 同时让多个 Agent 审查一个文档 |
| 一 Agent → 多人 | 最少见，Agent 主动向多人汇报 | Agent 在 Slack 频道通报状态 |

### 样式与品牌一致性

Every 为 Agent 配备了 Skills 文件来定义输出格式和品牌调性。如果你的 Agent 需要产出面向他人的内容，必须定义：

- 格式模板（数据如何呈现）
- 语言风格（正式/非正式，中文/英文）
- 品牌约定（命名、术语、视觉标准）

## 来自 Devin 的失败模式验证

Devin（Cognition 公司的 AI 软件工程师）是目前最大规模的"AI 员工"生产实践——服务 Goldman Sachs、Santander、Nubank 等企业，经历 18 个月迭代。其 2025 年度绩效报告中的失败模式，与 Claudie 的教训形成交叉验证，并揭示了更深层的边界：

### Agent 擅长的任务特征

| 特征 | 说明 |
|------|------|
| 需求明确 | 输入条件、预期输出、验收标准可精确描述 |
| 结果可验证 | 有客观标准判断"做对了没有"（测试通过、数据一致、格式合规） |
| 耗时但重复 | 人类工程师需要 4-8 小时的机械性工作 |

Devin 在这类任务上的表现：安全漏洞修复从每个 30 分钟降至 1.5 分钟（20x），代码迁移从 30-40 小时降至 3-4 小时（10x），PR 合并率从 34% 提升至 67%。

### Agent 的明确失败边界

**1. 模糊需求 = 必然失败**

> 视觉设计类任务必须提供"组件结构、颜色码、间距值"级别的精确描述。"做一个好看的登录页"不是可执行的指令。

这与 Claudie 的教训一致：Claudie 最初失败不是因为"不会做项目管理"，而是因为缺乏"在我们这里，好的项目管理具体长什么样"的精确定义。

**2. 中途变更需求导致性能下降**

Devin 发现"任务开始后继续追加指令通常导致性能下降"。Agent 不像人类那样能灵活 pivot——它基于初始指令构建执行计划，中途变更会打破内部一致性。

操作启示：**所有关键需求必须在任务启动前收集完毕**。Onboarding 阶段的信息收集不能留到执行阶段补。

**3. 无法处理人际判断**

Agent 无法管理利益相关者情绪、协调团队冲突、或在政治敏感的环境中做出社交性决策。这定义了 Agent 员工的硬边界——它能替代你做"事务"，但无法替代你做"关系"。

**4. 迭代式协作不如一次性交付**

人类习惯的"先做一版，我看看，然后调整"的协作模式在 Agent 身上效果不佳。更有效的模式是：**前期投入充分时间定义需求，然后让 Agent 一次性交付**，而非迭代修改。

### 与 Claudie 教训的映射

| Devin 的发现 | Claudie 的对应教训 |
|-------------|------------------|
| 模糊需求必然失败 | 岗位定义必须在构建之前完成 |
| 中途追加指令降低性能 | 手册必须预先定义完整 |
| 无法处理人际关系 | Agent 做事务，人做关系（Natalia 省下 15 小时用于与客户相处） |
| 需要人类写 playbook | 角色手册是 Agent 的全部认知基础 |

## 反模式

1. **"丢进去自己摸索"**：不定义岗位就开始构建，期望 Agent 像高级人类一样自己搞清楚该做什么。
2. **"一个 Agent 干所有事"**：忽视 context window 限制，让单个 Agent 处理过多信息。
3. **"发布即离开"**：构建完就不再维护手册和工具接入。McKinsey 在 50+ 个 Agentic AI 项目中确认，"发布即离开"是最常见的失败模式。
4. **"归咎于模型"**：Agent 表现不佳时，第一反应是"模型不行"而非检查自己的指令、架构和数据接入。
5. **"跳过信任阶梯"**：未经验证就赋予 Agent 大范围自主权。

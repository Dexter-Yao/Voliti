# 长时间运行Agent的上下文管理与弹性设计

## 一句话简介

长程 Agent 面临上下文耗尽、状态丢失、错误级联三大挑战，Anthropic 实证的"工程换班交接"模式——claude-progress.txt + 两部分架构 + 增量进展——是当前最可靠的工程实践。

## Takeaway

1. **Agent 会话 = 工程换班交接**：每次会话需要显式的状态文档和结构化任务边界
2. **三种上下文管理策略各有适用场景**：Compaction / Structured Note-Taking / Sub-Agent 按场景选择
3. **增量进展是关键**：一次只做一个功能，比一次性完成更可靠
4. **git commit 是状态持久化的最佳实践**：不仅保存代码，也是进展的客观记录
5. **三类失败模式必须防患于未然**：过早声明完成、半实现功能、未记录进展

## 摘要

### 上下文管理三策略

Anthropic 文档中记录的三个方向：

- **Compaction**：总结对话历史，压缩已处理信息
- **Structured Note-Taking**：在上下文外持久化记忆，以外部存储补充内部上下文
- **Sub-Agent Architectures**：每个专业 Agent 保持干净上下文，复杂任务通过组合完成

### 跨会话记忆架构（claude-progress.txt 模式）

Anthropic 实证的核心模式：

> "a claude-progress.txt file that keeps a log of what agents have done"

**两部分架构：**

> "an initializer agent that sets up the environment on the first run, and a coding agent that is tasked with making incremental progress in every session"

- Initializer agent：第一次运行时配置环境、建立上下文框架
- Coding agent：每次会话仅负责增量进展，从 progress file 读取上次状态

**增量进展策略（被验证为关键）：**

> "the next iteration of the coding agent was then asked to work on only one feature at a time. This incremental approach turned out to be critical"

**Git 最佳实践：**

> "ask the model to commit its progress to git with descriptive commit messages and to write summaries of its progress in a progress file"

Git commits 同时作为代码保存和进展记录，形成可追溯的状态历史。

### 三类失败模式

该架构明确应对的三个关键失败：

1. **过早声明完成**：Agent 声称任务完成但实际未完成
2. **半实现的功能**：功能部分实现但未达到可用状态，且未标记为进行中
3. **未记录的进展**：做了工作但没有写入 progress file，导致下次会话从错误状态开始

### 弹性与错误恢复

**何时重试：**
- 技术超时、短暂网络问题
- 策略：指数退避 + 随机抖动（避免惊群效应）

**何时升级（不再重试）：**
- 低置信度分类
- 高风险行动
- 监管合规影响
- 超出 Agent 能力范围的决策

**优雅降级架构：**
- 分层回退路径（主路径 → 备用路径 → 最小化路径）
- 冗余组件设计
- 断路器模式（Circuit Breaker）：检测持续失败后自动切断，防止级联扩散

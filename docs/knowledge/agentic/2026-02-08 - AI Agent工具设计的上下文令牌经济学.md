# AI Agent工具设计的上下文令牌经济学

## 一句话简介

从"把 API 包装成函数"到"工具设计是核心竞争力"的演变，以及 Tool Search 模式、Programmatic Tool Calling 等实测有效的上下文效率优化策略。

## Takeaway

1. **工具描述直接决定 Agent 行为**：工具文档被加载进 Agent 上下文，是"核心驱动"而非说明书
2. **少而精胜过多而杂**：大量工具等于大量上下文消耗，需要主动合并和下沉逻辑
3. **Tool Search 模式节省 85% 上下文**：~77K tokens → ~8.7K tokens，同时显著提升准确率
4. **"面向新人"写工具描述**：像向新同事解释一样描述工具，而非技术规格
5. **用评估驱动迭代**：工具优化需要 held-out test sets，不能凭直觉

## 摘要

### 工具设计哲学的演变

核心趋势：从"把 API 包装成函数"（2023）到"工具设计是核心竞争力"（2025-2026）。

Anthropic 的核心表述：

> "prompt-engineering your tool descriptions and specs" is "one of the most effective methods for improving tools" because descriptions are "loaded into your agents' context"

关键洞察：Anthropic SWE-bench agent "spent more time optimizing tools than the overall prompt"——工具优化投入回报高于提示词优化。

### 工具设计七原则

| 原则 | 说明 |
|------|------|
| **少而精** | 少量深思熟虑的工具胜过多而杂 |
| **语义合并** | 始终顺序调用的函数应合并为复合操作 |
| **逻辑下沉** | 将已知参数下沉到代码层面，减少 Agent 决策负担 |
| **高信号返回** | 返回 name, image_url 而非 uuid, mime_type |
| **防错设计** | 使用绝对路径、枚举而非自由文本 |
| **面向新人描述** | 像向"new hire on your team"解释一样写工具描述 |
| **评估驱动迭代** | 使用 held-out test sets 反复优化工具实现 |

**经验证的反模式：** Claude web search 工具中，模型无谓地在查询中附加年份，降低搜索质量——改进工具描述后问题消失。这证明行为问题的根源往往在工具描述，而非模型能力。

### 工具数量与上下文效率

**Tool Search 模式的实测效果：**
- 传统方式：暴露全部工具 → 消耗约 77K tokens
- Tool Search 方式：动态检索相关工具 → 仅消耗约 8.7K tokens（**节省 85%**）
- Opus 4 准确率：49% → **74%**
- Opus 4.5 准确率：79.5% → **88.1%**
- Tool Use Examples 将复杂参数处理准确率从 72% 提升到 **90%**

**Programmatic Tool Calling：** token 减少 **37%**（从 43,588 降至 27,297 tokens）。注：原调研声称"减少 80% 推理成本"有误，实际数据为 37% token 减少。

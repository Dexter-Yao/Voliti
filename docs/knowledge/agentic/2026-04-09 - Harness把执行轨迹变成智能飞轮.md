# Harness把执行轨迹变成智能飞轮

## 一句话简介

文章认为，当基模能力跨过阈值后，Agent 上限更多由 harness 决定，而 harness 捕获的执行轨迹正在取代静态语料，成为下一轮模型与产品竞争的关键数据资产。

## Takeaway

1. **Harness 已从辅助工程层上升为能力瓶颈层**：prompt engineering 解决“怎么说”，context engineering 解决“给什么信息”，harness engineering 则解决“如何把模型放进真实系统并稳定完成任务”。
2. **Harness 的核心不是多堆工具，而是围绕上下文预算构建信息、执行、反馈三层闭环**：信息层强调渐进式披露、精简工具和上下文隔离；执行层强调 research、plan、execute、verify 分段；反馈层强调把每次错误沉淀为永久约束与自动校准机制。
3. **“The Harness is the Dataset” 是本文最重要的产业判断**：谁能持续捕获高质量 execution traces，谁就更可能同时获得更强的后训练数据飞轮、更好的产品完成率，以及围绕 eval、trace、sandbox、workflow infra 的新价值捕获点。

## 摘要

### 为什么 Harness Engineering 变得重要

作者将 harness 比作驾驭模型这匹“野马”的马具，认为 AI 工程范式已经从 prompt engineering（单轮指令优化）进化到 context engineering（上下文供给优化），再进入 harness engineering（围绕模型构建完整运行与管控系统）的阶段。文章给出的定义是 `Agent = LLM + harness`：模型决定“做什么”，harness 决定“看见什么、能调什么工具、失败后如何处理”。

作者认为这一范式转移的底层逻辑是“瓶颈外移”：当模型能力过线，限制 Agent 上限的关键因素不再是基模本身，而是运行环境、工具系统、记忆管理、验证闭环与失败恢复机制。文中引用 Philipp Schmid 的判断：

> “The Harness is the Dataset.”

作者借此强调，竞争优势正在从参数和静态语料转向 harness 持续捕获的执行轨迹。

### Harness 的六个关键组件

文章将 harness 拆为六个组件：Memory & Context Management、Tools & Skills、Orchestration & Coordination、Infra & Guardrails、Evaluation & Verification、Tracing & Observability。作者进一步把这六个组件压缩成三层结构：

- 信息层：解决“当前时刻 Agent 应该看见什么”；
- 执行层：解决“任务如何被拆解、推进与协调”；
- 反馈层：解决“系统如何验证、复盘并持续变好”。

文章用 OpenClaw 作为例子说明，真正让 Agent 产生 “aha moment” 的并不是模型本身，而是 gateway、skills、memory、heartbeat 与 soul.md 等外围组件的组合。作者的核心论点是：很多单独看并不复杂的工程部件，一旦被组合成持续运行的系统，就会让 Agent 获得跨平台存在感、主动性与记忆积累能力。

### 信息层的设计原则

作者认为今天 Agent 的首要瓶颈仍是 context length，因此 harness 的很多设计原则都围绕上下文预算展开。

第一，渐进式披露。文章指出，OpenAI 早期尝试用巨大的 `AGENTS.md` 一次性写全规则，但后来发现这会挤占宝贵上下文。更可行的做法是按层组织信息：顶层的 `CLAUDE.md` / `AGENTS.md` 只保留元规则，`SKILL.md` 只在特定任务时按需加载，reference、脚本、日志等细节则在执行阶段再展开。

第二，工具要少而精。作者认为随着模型能力上升，外部工具数量不应线性增加；过多工具会带来决策瘫痪和幻觉空间。文中引用 Anthropic 与 Vercel 的实践，强调少数高复用“万能扳手”比庞杂工具库更可靠。

第三，寻找 context window 的甜蜜区间。文章认为更长上下文并不等于更好结果，原因在于 attention 会被稀释。作者引用长上下文检索测试，指出主流模型在 1M token 附近普遍出现明显下滑，因此许多顶级工程师会主动压缩上下文，把利用率控制在约 60% 以下。

第四，利用 subagent 做 context 隔离。作者引用 Boris Cherny 的 “context firewall” 说法，认为把查日志、读代码、搜文档等子任务拆给独立 subagent，可以减少主线程受到无关信息污染，让主线程只负责调度和收口。

### 执行层的设计原则

文章认为，很多 Agent 失败不是能力问题，而是任务结构设计错误。最重要的执行层原则是把 research、plan、execute、verify 刻意拆成独立阶段，而不是要求同一 session 一气呵成。作者认为研究与执行混在同一上下文里，会造成不必要的 context 污染。

文中引用 Boris Cherny 的规则：“任何非 trivial task 先进入 plan mode；如果事情偏离，就停止并重新规划。”作者进一步指出，人类应把精力从事后 code review 前移到 research 和 plan，因为糟糕计划通常会放大成大段糟糕执行。

### 反馈层与复利飞轮

文章认为反馈层是最容易被忽视、但最能产生复利的一层。作者引用 Mitchell Hashimoto 的定义：当 Agent 犯错时，应把时间投入到让系统永远不再犯这类错误的工程化方案中。无论是把错误经验写入 `AGENTS.md` / `CLAUDE.md`，还是通过自动校验、stop hook、浏览器自测与反复迭代，核心都是把单次失败转化为系统约束。

作者引用 Boris Cherny 的经验，认为只要提供有效验证手段，Claude 的最终质量通常可提升 2 至 3 倍。文章也引入 Karpathy 的 autoresearch 思路，强调重点不是让模型一次给出最佳答案，而是设计一个“提出假设→做实验→观察结果→保留有效策略→继续迭代”的可控闭环。

### 模型与 Harness 的关系

文章认为 “训练即部署” 是理解 harness 的关键。Agentic RL 与传统单轮 RLHF 不同，它依赖多轮、长链路、稀疏奖励、真实环境反馈和大量工具调用。因此，训练效果在很大程度上由训练环境是否贴近真实世界决定。作者用 Cursor 与 Windsurf 的例子说明：模型在接近真实 coding harness 的环境中训练后，会自发学会更深的代码检索、顺手修 linter、补测试、减少大改等行为。

在这个框架下，harness 不再只是模型外面的脚手架，而是模型能力形成的土壤。文章进一步指出，许多原本属于 harness 的能力，例如 tool search、多步工具调用、compaction、summarization、planning、self-verification，正在被模型厂逐步吸收进后训练流程。因此 harness 是一个一边创造价值、一边把自身经验输送回模型内部的奇特层。

### 创业机会与下一阶段

文章把潜在创业机会大致分布在三层：信息层的企业 context 抽取与流程还原，执行层的 workflow orchestration、durable execution、security/governance 与 sandbox，反馈层的 evaluation 与 observability。作者用 Edra、Temporal、Oasis Security、Daytona、Braintrust 等公司说明：围绕 context、执行环境、权限控制、trace 与 eval 的基础设施层，正在形成新的独立价值点。

在展望部分，作者提出 harness engineering 之后的下一阶段可能是 coordination engineering，即围绕多 Agent / 多人节点协作的组织控制面建设。文章认为，未来 AI 产品不一定只是“更聪明的单个 Agent”，更可能是一个可视化监工面板与协作网络，其任务是管理多节点的状态、授权、介入与收口。

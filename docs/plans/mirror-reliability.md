# Mirror 展示层稳定性收口设计

## 一、文档定位

本文档定义 Mirror 页在当前分支中的目标形态、边界条件、复用路径、失败处理、测试范围与实施约束。

本文档只覆盖 iOS 客户端 Mirror 展示层，不覆盖以下内容：

1. Coach 对话协议扩展。
2. backend Store schema 变更。
3. Journal 与 Mirror 的长期统一重构。
4. 新的通用展示基础设施或 section 范围系统。

## 二、问题定义

Mirror 当前骨架方向正确，但展示语义仍然混杂，主要体现在以下几个方面：

1. 事件流仍保留“最近 N 条”分页骨架，无法表达“当前范围内的完整日志”。
2. 空状态、筛选后空状态、stale 异常态之间的语义边界不够清楚。
3. 指标首页与历史页对 `estimated` 质量标记、单位与格式的处理尚未统一。
4. 日志、指标、LifeSign 对时间范围的解释容易混成两套口径。

Mirror 的目标不是做成一张更花哨的报表，而是成为一个可信的长期镜面。用户需要在这里看到的是身份、状态、证据、质量边界，以及当数据不可靠时系统如何诚实地承认这一点。

## 三、目标与非目标

### 3.1 目标

1. 保持 `Chapter / 北极星 / 支持指标 / LifeSign` 作为全局长期镜面。
2. 将 `LOG / 日志` 明确收口为唯一受日期范围控制的区域。
3. 在不扩协议的前提下，清楚展示事件、证据、质量边界与异常恢复路径。
4. 清理旧分页语义与兼容分支，不保留双轨行为。
5. 通过最小抽象完成统一显示规则与测试补齐。

### 3.2 非目标

1. 不把日期范围扩展到整个 Mirror 页面。
2. 不把日志时间范围写入共享 Store。
3. 不新增服务层、注册表或通用 section scope 基础设施。
4. 不将事件流升级为更重的卡片体系。
5. 不调整 Coach 会话页面与对话协议。

## 四、现有代码复用基线

当前方案应直接建立在以下实现之上，而不是平行重写：

1. `frontend-ios/Voliti/Features/Mirror/MirrorViewModel.swift`
   - 已持有 `BehaviorEvent` 聚合、事件分组、筛选骨架与 Mirror 主状态。
   - 当前问题是保留了“最近 N 条”的分页语义，需要被移除或重写为日志范围语义。
2. `frontend-ios/Voliti/Core/Data/MetricComputer.swift`
   - 已提供 `reported / estimated / missing` 的基础事实来源。
   - 继续作为指标与趋势质量判断的唯一来源。
3. `frontend-ios/Voliti/Core/Network/StoreSyncService.swift`
   - 已提供 `ProjectionFreshness` 与 `syncAll()`。
   - stale 恢复动作必须建立在这一套基础上。
4. `frontend-ios/Voliti/Core/Data/Models/BehaviorEvent.swift`
   - 已具备 `evidence`、`metrics`、`refs` 等字段。
   - 足以支撑日志区证据展示，不需要扩协议。
5. `frontend-ios/Voliti/Features/Journal/EventRow.swift`
   - 继续作为日志项共享底座。
   - Mirror 只做轻量层级包装，不 fork 一套事件行。

## 五、目标信息结构

Mirror 页目标结构如下：

```text
Mirror
├── Chapter
├── North Star
├── Support Metrics
├── LifeSign
└── LOG / 日志
    ├── 标题
    ├── 日期范围入口（右侧）
    ├── 类型筛选
    ├── 范围内日志列表
    └── 空态 / stale / 恢复动作
```

### 5.1 布局原则

1. 上半部分是长期镜面，不随日志时间范围变化。
2. 日志区是浏览器，时间范围与筛选都只影响该区。
3. 日期范围入口放在 `LOG / 日志` 标题右侧，默认显示预设标签，`自定义` 时显示完整日期范围。

## 六、关键架构决定

### 6.1 时间范围边界

1. 日期范围只作用于日志区。
2. `Chapter` 不受日志日期范围影响。
3. `North Star`、支持指标、`LifeSign` 维持全局摘要语义。
4. `LifeSign` 不跟随日志日期范围，否则会在同一页形成双重时间口径。

### 6.2 本地持久化边界

1. 日期范围是设备本地 UI 偏好。
2. 日期范围持久化在客户端本地，不进入共享 Store。
3. 客户端允许恢复上一次日志范围选择，但不得把该偏好误当作共享业务状态。

### 6.3 数据查询与投影边界

1. 日志区切换范围时，必须查询该范围内全部匹配的 `BehaviorEvent`。
2. 不再保留“最近 N 条”的旧分页逻辑。
3. 真相层是“当前范围内全量事件”，UI 层用 `LazyVStack` 与按天折叠控制渲染成本。
4. 筛选计数、分组列表、空状态均基于同一份范围事件集投影，不允许各自再取不同口径。
5. 当前阶段不为远期超大时间范围预埋额外聚合或增量投影基础设施，优先保持实现直接与可验证。

### 6.4 展示职责划分

1. `MirrorViewModel` 负责语义状态收口。
2. 共享 helper 负责指标值、单位、质量标记、历史页格式的一致性。
3. View 只负责渲染，不复制业务判断。

### 6.5 当前工程收口决定

基于当前分支的 eng review，以下实现路径已锁定：

1. 日志范围、筛选、loading、空态与应用逻辑统一收口在 `MirrorViewModel`。
2. `MirrorView` 只保留本地 UI 偏好读取与范围选择 sheet 呈现，不承担日志业务判断。
3. 旧“最近 N 条”分页语义必须直接删除，不保留 fallback、旁路或无范围默认分页模式。
4. stale 恢复动作直接复用 `StoreSyncService.syncAll()`，不新增 Mirror 专属同步层，也不允许“只重算本地缓存”的伪恢复路径。

## 七、交互设计

### 7.1 日志范围入口

入口形态如下：

```text
LOG / 日志                          近30天 ▾
```

或在自定义范围下：

```text
LOG / 日志          2026.04.01 - 2026.04.11 ▾
```

### 7.2 选择器流程

```text
点击范围入口
  → 打开轻量 sheet
    → 预设项：7天 / 30天 / 90天 / 本篇章 / 自定义
      → 预设项单击即生效并关闭
      → 自定义进入日期选择
        → 用户确认“应用”后才生效
        → 取消时保留当前范围
```

约束如下：

1. `本篇章` 在无 chapter 时显示为禁用态并带说明。
2. 自定义结束日期不得晚于今天。
3. 自定义非法范围必须显式拒绝，不允许 silent correction。

## 八、状态模型

### 8.1 页级状态

Mirror 页仍保留页级 fresh / stale 语义。

```text
Mirror Page
├── fresh
└── stale
    ├── 显示友好提示
    ├── 禁用日志范围切换
    └── 提供“刷新数据”动作
```

### 8.2 日志区状态

```text
LOG / 日志
├── loading
├── ready
│   ├── 全部
│   └── 已筛选
├── empty-in-range
├── empty-after-filter
└── range-apply-failed
```

状态语义如下：

1. `loading`
   - 切换范围期间显示显式加载态。
   - 不得继续展示旧日志冒充新范围结果。
2. `empty-in-range`
   - 当前范围内确实没有日志。
   - 文案只描述该范围没有记录。
3. `empty-after-filter`
   - 当前范围内有日志，但当前筛选条件没有命中。
   - 文案必须明确是“当前筛选条件下暂无记录”，并提供“一键查看全部”。
4. `range-apply-failed`
   - 新范围应用失败时回退旧范围。
   - 用户看到恢复后的旧范围，并收到明确错误提示。

## 九、指标与质量标记

### 9.1 唯一事实原则

指标首页与历史页必须复用同一套显示规则，至少统一以下内容：

1. 值格式。
2. 单位。
3. 比例类分母处理。
4. `estimated` 质量标记。
5. `missing` 的空态表达。

### 9.2 质量标记范围

1. 质量标记主要用于指标与历史趋势。
2. 事件流保留 `summary + evidence`，不额外引入事件级质量芯片。
3. 趋势图可继续以透明度等轻量方式表达 `estimated`。

## 十、错误与恢复表

| 场景 | 失败方式 | 系统处理 | 用户看到的结果 |
|------|----------|----------|----------------|
| 打开范围入口 | 本地状态异常 | 保留当前已生效范围 | 范围不变 |
| 应用预设范围 | 范围状态写入失败 | 不应用新范围并记录日志 | 范围不变，日志不假切换 |
| 自定义范围校验 | 起止日期非法、结束早于开始、结束晚于今天 | 显式拒绝 | 需要修正后再应用 |
| 应用自定义范围 | 日志重算失败 | 回退旧范围并记录失败 | 保留旧日志视图 |
| stale 刷新 | `syncAll()` 失败 | 保持 stale，允许再次尝试 | 仍为异常态，但知道失败 |
| 保留筛选切换范围 | 新范围内当前筛选为 0 | 进入筛选后空态 | 提供“一键查看全部” |

## 十一、失败模式与关键缺口

以下项目属于本次必须消除或补测的关键缺口：

1. 日志范围切换后仍显示旧日志，未进入 loading。
2. 自定义范围非法却被静默修正。
3. stale 时仍允许切换日志范围。
4. 筛选后空态误用“该范围内暂无记录”。
5. 旧“最近 N 条”分页逻辑残留并污染新日志范围语义。
6. 首页与历史页对 `estimated` 的处理不一致。

## 十二、可观测性

最低可观测性要求如下：

1. 目标日志范围。
2. 当前生效日志范围。
3. 范围内事件总数。
4. filter 后事件数。
5. projection build success / fail。
6. stale refresh success / fail。

说明：

1. 这些日志只为排障服务，不进入用户界面。
2. 界面文案不得暴露“本地缓存重算”等技术术语。

## 十三、测试策略

### 13.1 测试重点

本次以纯逻辑测试为主，覆盖以下分支：

1. 日志范围切换。
2. 预设范围即时应用。
3. 自定义范围显式应用。
4. 非法范围拒绝。
5. 保留筛选后切换范围。
6. `empty-in-range` 与 `empty-after-filter` 区分。
7. stale 禁用范围切换。
8. stale CTA 触发恢复成功 / 失败。
9. 指标首页与历史页的统一格式与质量标记。

### 13.2 UI 测试策略

只保留一条最小 smoke：

```text
切换日志范围
  → 应用
    → 日志区进入 loading
      → 渲染新范围结果
```

本次不引入一组新的 SwiftUI snapshot 测试。

### 13.3 UI Smoke

当前分支补一个最小 `VolitiUITests` target，仅承载一条真实交互 smoke：

```text
打开 Mirror
  → 点击日志范围入口
    → 选择预设范围
      → 日志区进入 loading
        → 渲染新范围结果
```

约束如下：

1. 不把这次 UI smoke 扩展成新的大规模 UI 测试工程。
2. 其余覆盖仍以 `VolitiTests` 中的逻辑与集成测试为主。

## 十四、实施边界

### 14.1 in scope

1. `MirrorViewModel` 的日志范围、状态收口与旧分页清理。
2. `MirrorView` 与日志区范围入口交互。
3. `EventStreamSection` 的空态、筛选后空态与日志渲染收口。
4. 指标显示 helper 的最小提炼。
5. `MetricHistoryView` 与首页指标语义对齐。
6. stale 友好提示与刷新动作接入。
7. 对应逻辑测试与一条 UI smoke。

### 14.2 not in scope

1. backend Store schema 变更。
2. Coach 协议或 Coach 页面变更。
3. Journal 与 Mirror 的统一浏览系统。
4. 跳转到某一天或月份锚点能力。
5. 范围内日志导出或分享。
6. Mirror 专属同步服务、通用 range coordinator 或 section scope 基础设施。

## 十五、工作分解建议

```text
Lane A
  MirrorViewModel
    → 日志范围状态
    → 旧分页清理
    → stale 处理

Lane B
  MirrorView / EventStreamSection
    → 范围入口
    → 空态
    → loading / stale 交互

Lane C
  Metric helper / MetricHistoryView
    → 格式统一
    → quality 对齐

Lane D
  Tests
    → ViewModel 逻辑
    → helper
    → smoke
```

当前阶段可以并行的前提是先锁定日志范围状态模型与指标 helper 的命名。否则 B、C、D 都会反复改。

## 十六、当前阶段工程取舍

本次实现遵循当前产品阶段约束：

1. 允许日志范围查询在较大时间窗下直接拉取该范围内全量事件。
2. 不为“产品上线一年后可能出现的超大事件量”提前引入更重的聚合、缓存或数据库层优化。
3. 一旦后续真实用户量和日志体量证明这块成为瓶颈，再按实际数据回头优化，而不是在当前阶段预支复杂度。
